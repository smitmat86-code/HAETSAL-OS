## What We're Building

Expose semantic memory recall through the canonical MCP interface instead of as
an engine-specific Hindsight product concept.

Session 7.1 made Hindsight a canonical projection target. Session 7.2 uses that
projection state to provide a canonical semantic read mode that:

- queries Hindsight through the existing HAETSAL runtime
- returns results through the canonical MCP contract
- includes source/provenance linkback to canonical records where possible
- surfaces engine/projected status truthfully through canonical memory status

When complete, callers should ask HAETSAL for canonical semantic memory reads,
not for “Hindsight recall” directly. The Hindsight engine remains an internal
projection implementation detail behind the canonical open-brain interface.

---

## THE Brain Laws Check

### Law 1: One Public Face

All semantic recall remains behind the existing McpAgent/Worker interface. This
session may rewire canonical query services and tool outputs, but it must not
introduce a public Hindsight endpoint or a second engine-specific read route.

### Law 2: Zero-Knowledge

Canonical/Hindsight read results must continue to respect the current
zero-knowledge posture. No raw memory content may be copied into D1, KV, or
Analytics Engine for recall convenience. Provenance and linkback metadata may be
stored in D1, but the underlying content must still come from the proper memory
systems and existing trusted runtime.

### Law 3: Agent Write Policy

This session concerns reads, not writes. It must not widen any write path or
introduce a path to persist procedural memory through semantic recall
bookkeeping. Any optional read-side status caching must remain metadata-only.

### Authorization

| Route / Entry Point | Auth Level | Tenant ID Source | When Checked |
|---------------------|-----------|-----------------|-------------|
| Canonical semantic read tools via McpAgent | CF Access session | Existing McpAgent tenant context | Existing auth middleware + DO session bootstrap |
| Internal canonical semantic query service | Trusted internal caller | Service arg / tenant context | At service boundary |
| Canonical memory status lookup | Trusted internal caller / McpAgent | Service arg / tenant context | At service boundary |

No intentionally public route is added in this spec.

### Action Layer (if applicable)

No action tools in this spec.

### Audit

| Event | Table | Action String | Blocking? |
|-------|-------|-------------|-----------|
| Semantic read request dispatched | existing audit lane if already present for read telemetry | existing read action string only | Non-blocking / existing behavior |
| Canonical semantic read fallback due to missing projection | memory / canonical audit lane only if current read telemetry already exists | metadata-only | Non-blocking |
| Canonical memory status read | existing status/read telemetry only | metadata-only | Non-blocking |

This session should not introduce noisy new audit writes for every recall unless
they fit the current HAETSAL read telemetry pattern. If no current audit exists
for read-path queries, do not invent a new high-volume audit stream just for
this session.

### Compute Routing

| Operation | Tier | Rationale |
|-----------|------|-----------|
| Canonical semantic recall orchestration | Sync Worker / service | MCP read contract |
| Hindsight semantic query | Sync Worker / existing transport | Existing Hindsight recall runtime |
| Canonical provenance/status join | Sync Worker / D1 query | Lightweight metadata join |
| Reflect / consolidation changes | Deferred to Session 7.3 | Out of scope |
| Graph / timeline query expansion | Deferred to 8.x | Out of scope |

### Other Behavioral Wiring

- The public MCP contract should expose semantic recall as canonical memory
  behavior rather than as “Hindsight” behavior.
- Query orchestration should prefer canonical metadata + Hindsight projection
  references over bespoke engine-specific state.
- Returned results should include enough provenance to answer:
  - which canonical capture/document/source produced this memory
  - whether the result is semantic/Hindsight-backed
  - what canonical status/result row it came through when available
- If some canonical captures have not yet produced Hindsight projections, the
  behavior must be truthful. Use the smallest fallback that preserves a stable
  contract:
  - either exclude not-yet-projected items from semantic mode
  - or return them with explicit non-semantic status
  But do not pretend semantic projection exists when it does not.

### `src/services/canonical-semantic-recall.ts`

Responsibilities:

- orchestrate a semantic recall request through the canonical interface
- resolve which Hindsight-backed canonical projections are eligible
- call the existing Hindsight recall runtime
- normalize engine-specific results into the canonical semantic result shape
- attach provenance/linkback metadata

- do not expose raw Hindsight response objects directly through MCP
- do not bypass the canonical metadata layer when provenance can be resolved

### `src/services/canonical-semantic-linkback.ts`

Responsibilities:

- map Hindsight result references back to canonical capture/document metadata
- attach canonical operation/projection status context where available
- keep joins small and metadata-only

If existing canonical status/read-model services already provide most of this,
prefer reusing them instead of building a duplicate mapper.

### `src/services/canonical-memory-status.ts`

Responsibilities:

- extend status reporting as needed so semantic reads can reflect truthful
  Hindsight projection readiness

This session should not redesign status wholesale; it should only add what is
needed for semantic recall and provenance visibility.

2. Canonical query layer selects semantic mode

## As-Built Record

**Built**
- `src/services/canonical-semantic-recall.ts` adds the canonical semantic-query
  branch that calls the existing Hindsight recall runtime, normalizes results
  into the canonical search shape, and returns a truthful `status` of `ok`,
  `partial`, or `unavailable`.
- `src/services/canonical-semantic-linkback.ts` resolves Hindsight recall
  identifiers back onto canonical projection results, capture/document ids, and
  canonical operation metadata without storing any raw memory content in D1.
- `src/services/canonical-memory-query.ts` now routes `search_memory` to the
  semantic branch when `mode = 'semantic'` and preserves the existing lexical
  read path as the default.
- `src/services/canonical-memory-status.ts` now exposes `resultStatus`,
  `projectionResultId`, `engineDocumentId`, `engineOperationId`, and
  `semanticReady` for truthful Hindsight-backed status visibility.
- `src/tools/canonical-memory.ts` extends `search_memory` with a canonical
  `mode: 'lexical' | 'semantic'` selector instead of adding a new engine-facing
  tool.
- `src/types/canonical-memory-query.ts` extends the canonical search/status
  contracts with semantic result metadata (`recallText`, `provenance`,
  `semanticStatus`) while keeping the legacy lexical fields intact.
- `src/types/hindsight.ts` was updated to match the runtime request/response
  shapes already used by the HAETSAL shell so semantic recall normalization can
  rely on typed Hindsight recall results.
- `tests/7.2-semantic-recall-through-canonical-interface.test.ts` adds note,
  conversation, mixed-linkback, missing-projection, status, and engine-failure
  coverage through the canonical MCP tool surface.

**Migration Decision**
- `1015_semantic_recall_linkback.sql` was **not needed**.
- Session 7.2 reused the existing 6.1 + 7.1 schema by joining against
  `canonical_projection_results.engine_document_id`,
  `canonical_projection_results.engine_operation_id`, and `target_ref`.

**Canonical Semantic Result Shape Shipped**
- `search_memory` now accepts `mode?: 'lexical' | 'semantic'`.
- `search_memory` now returns:
  - `query`
  - `mode: 'lexical' | 'semantic'`
  - `status: 'ok' | 'partial' | 'unavailable'`
  - `items[]`
- In semantic mode, each item may additionally include:
  - `mode: 'semantic'`
  - `recallText`
  - `provenance` with `projectionKind`, canonical capture/document/operation
    ids when available, projection job/result ids when available, `targetRef`,
    and `sourceSystem`
  - `semanticStatus` with `projectionStatus`, `resultStatus`, and `ready`
- Existing lexical/default reads still return the same canonical preview-style
  items, now tagged with `mode: 'lexical'`.

**Provenance / Linkback Implementation**
- Hindsight recall results are matched back to canonical records by checking the
  runtime result for `document_id`, `source_document_id`, `target_ref`, or
  related metadata and joining that against the stored 7.1 projection-result
  references.
- When a canonical match exists, the semantic result returns the canonical
  capture id, canonical document id, canonical operation id, and the specific
  projection job/result ids that backed the semantic answer.
- When a recall result cannot be linked back truthfully, the result is still
  returned as semantic output, but the canonical ids stay `null` and
  `semanticStatus.ready` stays `false`.

**Explicit Deviations**
- The spec allowed excluding not-yet-projected captures or returning them with
  explicit non-semantic status. The shipped behavior takes the smaller path:
  semantic mode excludes not-yet-projected captures because Hindsight has not
  materialized them yet, while default lexical mode still finds the canonical
  record.
- On Hindsight semantic-recall failure, the shipped canonical contract returns
  `mode: 'semantic'`, `status: 'unavailable'`, and `items: []` rather than
  throwing raw engine errors or silently substituting lexical search results.
- `memory_status` gained a few extra projection fields beyond the sketch in the
  spec so callers can inspect truthful semantic readiness without needing a new
  status tool or a second D1 lookup.
