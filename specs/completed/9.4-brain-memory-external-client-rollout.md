## What We're Building

Roll out the first real external-client surface for the open brain:

- `brain-memory`

This is the scoped surface that MCP-native clients use to both:

- **write to the brain**
- **read from the brain**

It is intentionally narrower than "all of HAETSAL." It is the safe,
memory-oriented connection point for clients such as:

- Codex
- Claude Code
- Cursor
- similar MCP-native development/operator clients

This session turns the architecture from 9.3 into a concrete first rollout by
defining:

- what `brain-memory` contains
- what ingestion patterns MCP-native clients should use
- what it explicitly does **not** include
- how clients should capture durable value without pretending to have
  "unlimited context"

This is not the Google connector session and not the Chief-of-Staff session.

---

## Why This Session Exists

The brain is now ready enough that external clients should stop treating memory
as ad hoc prompt stuffing and start using a stable shared surface.

But if we jump straight to "connect Codex/Claude/Cursor," we risk confusion
about what they are actually connecting to.

The answer is:

- not directly to Hindsight
- not directly to Graphiti
- not to Google
- not to outbound action tools

They connect first to **`brain-memory`**, which provides:

- explicit memory capture
- canonical query / recall
- graph/timeline reads where appropriate
- artifact- and session-based memory compounding

The goal is durable, portable, retrievable context across clients, not
full-transcript omnivorous logging.

---

## THE Brain Laws Check

### Law 1: One Public Face

`brain-memory` must remain part of the existing canonical public brain story.
This session may narrow the exposed tool set for external MCP-native clients,
but it must not create a second brain or a client-specific shadow backend.

### Law 2: Zero-Knowledge

Client rollout must not introduce new raw-content caches or session transcript
dumps into D1, KV, Analytics Engine, or rollout-side stores. Canonical capture
and R2/Postgres remain the durable truth.

### Law 3: Agent Write Policy

This session may expose memory capture/write and read/query behavior only. It
must not broaden source-read or source-write authority as part of the first
external client rollout.

---

## Strategic Answer

### Does `brain-memory` handle both read and write?

Yes.

`brain-memory` is the scoped surface for both:

- writing durable memory into the brain
- reading memory back out of the brain

### What is the default external-client memory posture?

Selective, structured capture plus retrievable context.

Not:

- full transcript mirroring
- every prompt/response pair as durable memory
- unlimited always-in-prompt context

### What should compounding memory come from?

The first rollout should support three durable patterns:

1. explicit capture
2. session-close summary capture
3. artifact-linked capture

These create compounding memory without flooding the system with low-signal
noise.

## Behavioral Wiring

### Integration Surface

This session rolls out only:

- `brain-memory`

It does **not** include:

- `brain-sources-read`
- `brain-actions`

### Client Classes

| Client | Initial Connection Pattern | Initial Scope |
|--------|-----------------------------|---------------|
| Codex | Remote MCP | `brain-memory` |
| Claude Code | Remote MCP | `brain-memory` |
| Cursor / similar | Remote MCP | `brain-memory` |

This session should define the connection pattern in a brand-neutral way.
Differences between clients should be operational/configurational, not
architectural.

### Allowed Operations

Allowed through `brain-memory`:

- explicit memory capture
- search/query memory
- recent memories
- document lookup
- memory status
- memory stats
- semantic recall
- graph/timeline reads
- context-prep reads when already part of the canonical memory surface

Forbidden in this rollout:

- Gmail / Calendar / Drive read
- email send / draft / archive
- calendar event create/update
- document/task system mutation
- arbitrary file-system memory scraping outside explicit capture paths

### Ingestion Modes

Define three first-class ingestion modes for MCP-native clients.

#### 1. Explicit Capture

Use when the client or user intentionally wants to retain something durable.

Examples:

- decision made
- preference learned
- project finding
- insight / plan / rule of thumb

#### 2. Session-Close Summary Capture

Use as the default compounding mechanism for working sessions.

Examples:

- what changed
- key decisions
- unresolved questions
- next steps
- notable user preferences revealed in-session

This should be the preferred default over full transcript retention.

#### 3. Artifact-Linked Capture

Use when a durable artifact is the real thing worth remembering.

Examples:

- spec
- plan
- doc
- code-review outcome
- milestone note

The brain should capture the durable meaning and provenance of the artifact,
not just blind raw duplication.

### Auth / Policy Direction

Current Access-gated HAETSAL auth remains acceptable for the first rollout.

Future modernization toward more OAuth-native remote MCP posture is allowed
later, but should not block the first `brain-memory` rollout.

## Part 3: Types / Contract

The rollout contract should define shapes along these lines:

```typescript
export type ExternalClientCaptureMode =
  | 'explicit'
  | 'session_summary'
  | 'artifact'

export interface ExternalClientCaptureInput {
  mode: ExternalClientCaptureMode
  content: string
  scope?: string | null
  provenance?: string | null
  artifactRef?: string | null
}

export interface BrainMemorySurfaceProfile {
  canCapture: boolean
  canQuery: boolean
  canReadSources: false
  canMutateSources: false
}
```

Exact names may differ, but the contract must answer:

- how does a client declare what sort of capture it is doing?
- what does this rollout surface allow?
- what does it explicitly not allow?

If the current canonical capture tool can support this with additive metadata
rather than a new public tool, prefer that.

## Part 4: Services

### `src/services/external-client-memory.ts`

Responsibilities:

- normalize client-side capture modes into canonical memory calls
- enforce rollout scope (`brain-memory` only)
- provide a clean profile for MCP-native clients

Responsibilities it must **not** take on:

- source-system OAuth
- Gmail/Calendar/Drive fetches
- external actions
- BYOC export/import

### Retrieval posture

The read side should reuse:

- `search_memory`
- `get_recent_memories`
- `get_document`
- `memory_status`
- `memory_stats`
- graph/timeline and composed reads where already available through the
  canonical surface

Do not invent a second retrieval pathway just for external clients.

---

## Part 5: MCP / Tool Surface

Preferred posture:

- keep using the canonical memory tool family
- annotate / constrain it as the first `brain-memory` rollout surface

Possible implementation choices:

### Option A: No new public tool names

Reuse:

- `capture_memory`
- `search_memory`
- `get_recent_memories`
- `get_document`
- `memory_status`
- `memory_stats`

Add rollout-safe capture-mode metadata only.

### Option B: One narrow helper

Add a thin helper such as:

- `capture_client_memory`

only if it meaningfully reduces ambiguity for MCP-native clients.

Default recommendation:

- prefer **Option A**
- keep one canonical memory contract
- add the smallest possible rollout-specific affordances

---

## Part 6: What This Is Not

This session is explicitly **not**:

- the Google read-only ingestion rollout
- the BYOC / web-client bridge
- the Chief-of-Staff context builder
- the outbound action surface

Those should come later as separate sessions.

This means 9.5 can be done in parallel, but it should assume:

- `brain-memory` already owns memory-only client read/write
- Google belongs under `brain-sources-read`
- Google read-only ingestion is selective and provenance-rich
- Google write/actions remain later

## Definition of Done

- HAETSAL has a real first external-client rollout surface for MCP-native
  clients.
- That surface clearly handles both writing memory and reading memory.
- It compounds durable context through explicit capture, session summaries, and
  artifact-linked capture.
- It avoids the trap of "store everything forever."
- It sets up 9.5 cleanly without blending memory-only access with source
  connectors.
