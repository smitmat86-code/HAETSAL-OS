## What We're Building

Finalize the deployment posture and canonical graph projection contract for
Graphiti before any live Graphiti ingestion work begins.

Phase 7 established the semantic/Hindsight side of the advanced open brain.
Session 8.1 does the equivalent design work for the graph/temporal side so that
Session 8.2 can implement Graphiti fan-out without guessing about:

- where Graphiti should run
- how canonical captures map to episodes/entities/edges
- how Graphiti identities reconcile back to canonical records
- how graph projection lifecycle should appear in the canonical operation model

This is a design-and-contract session. It should produce the smallest concrete
implementation seam needed for 8.2, not a full Graphiti ingestion worker.

---

## THE Brain Laws Check

### Law 1: One Public Face

This session must not introduce a new public Graphiti route or bypass the
McpAgent/Worker surface. Any new code should define internal contracts, typed
projection shapes, and status/design surfaces only.

### Law 2: Zero-Knowledge

Graphiti design work must preserve the rule that canonical capture remains the
source of truth and that raw memory content is not denormalized into D1, KV, or
Analytics Engine for graph convenience. If the design requires Graphiti to see
content, that must happen through the same trusted projection-runtime boundary
used for Hindsight-style engines, not through new operational metadata copies.

### Law 3: Agent Write Policy

This session changes graph projection design, not write policy. It must not
create any new direct path for clients or agents to write only to Graphiti, and
it must not widen procedural-write permissions.

## Part 3: Types / Contract

The design should define a stable contract along these lines:

```typescript
export interface CanonicalGraphProjectionInput {
  tenantId: string
  captureId: string
  documentId: string
  operationId: string
  scope: string
  sourceSystem: string
}

export interface GraphProjectionEpisode {
  canonicalCaptureId: string
  graphEpisodeRef?: string | null
  kind: 'note' | 'conversation' | 'artifact' | 'event'
}

export interface GraphProjectionEntity {
  canonicalKey: string
  graphEntityRef?: string | null
  label: string
  type?: string | null
}

export interface GraphProjectionEdge {
  fromCanonicalKey: string
  toCanonicalKey: string
  relation: string
  validAt?: number | null
}
```

Exact names may differ, but the contract must answer:

- what constitutes an episode?
- how are entities identified across captures?
- when is an edge created or updated?
- how do temporal updates reconcile with prior graph state?

## Part 4: Design Decisions Required

Session 8.1 must make the smallest explicit decision on each of these:

### 1. Deployment Posture

Pick the initial Graphiti runtime posture and document why.

Expected options:

- Cloudflare Containers first
- external service first, Cloudflare-orchestrated
- staged posture: external first, Cloudflare-hosted later

The chosen posture should optimize for:

- architectural coherence with the advanced open-brain target
- minimum implementation risk for Session 8.2
- provenance/control compatibility with canonical capture

### 2. Projection Mapping Model

Define how canonical capture shapes project into graph constructs.

Minimum required mapping:

- note capture -> episode + entity candidates + relationship candidates
- conversation capture -> episode + speaker/subject entities + temporal edges
- artifact capture -> episode + artifact/entity references

### 3. Reconciliation Rules

Define the rules for:

- entity re-identification across captures
- edge deduplication or merge behavior
- temporal updates versus replacement
- how graph refs map back to canonical ids

### 4. Canonical Status Surface

Define how graph projection lifecycle should eventually appear in canonical
status/operations without implementing the full 8.2 runtime yet.

## Part 7: Implementation Notes

- Favor a staged deployment choice if that reduces risk and still preserves the
  long-term architecture.
- Keep the Graphiti contract parallel to the Hindsight projection contract
  where that improves coherence, but do not force false symmetry if graph
  semantics need different reconciliation rules.
- Preserve the principle that clients should not need to know whether a future
  answer came from Hindsight, Graphiti, or canonical source unless they ask.
- Do not broaden into:
  - live Graphiti projection fan-out
  - graph query tools
  - timeline traversal tools
  - speculative multi-engine ranking or merge logic

### Chosen Graphiti Deployment Posture

- **Staged posture: external Graphiti service first, Cloudflare-hosted later.**

- Session 8.2 should submit Graphiti work only through the trusted internal

### Projection Mapping Model Shipped

- `CanonicalGraphProjectionInput` is metadata-only and carries canonical ids,
  scope, source system, and source ref.
- `buildCanonicalGraphProjectionPlan()` produces:
  - one canonical episode per capture
  - deterministic anchor entities for scope and source on every capture
  - title-derived topic entities when available
  - conversation speaker entities plus temporal conversation edges
  - artifact/document entities plus artifact backing edges for file-style input
- The design contract keeps raw body access inside the future trusted Graphiti
  runtime only and explicitly forbids operational metadata denormalization.

### Reconciliation Rules Shipped

- Entity anchors reconcile by stable `canonicalKey`.
- Scope/source/document/artifact anchors use `identityStrategy =
  canonical_anchor`.
- Stable literals like topic titles and generic speakers use
  `identityStrategy = stable_literal`.
- Structural edges deduplicate by relation plus endpoints.
- Temporal edges use `append_valid_time` and append observations instead of
  replacing prior relationship history.
- Future Graphiti refs must map back through canonical episode/entity/edge
  anchors rooted in canonical capture/document/operation ids.

### Canonical Status Surface Shipped

- `memory_status` now includes a top-level `graph` section:
  - `mode: 'graphiti'`
  - `status: 'pending' | 'queued' | 'projected' | 'failed'`
  - `ready`
  - `jobId`
  - `projectionResultId`
  - `targetRef`
  - `errorMessage`
  - `updatedAt`
  - `reconciliationMode`
- This is contract-only in 8.1. It reflects the existing canonical
  projection-job/result rows without implementing Graphiti ingestion.
