## What We're Building

Build the first canonical **context assembly layer** for a first-party HAETSAL
agent: the Chief of Staff.

Session 9.1 gives HAETSAL a multi-mode router that can choose raw, semantic,
graph, or composed retrieval. Session 9.2 turns that retrieval capability into
a stable high-value product primitive:

- `prepare_context_for_agent`

This is not a general action orchestrator and not a full Chief-of-Staff agent
workflow. It is the context-builder that assembles the right information bundle
before a first-party agent thinks, plans, or acts.

The output should help a Chief of Staff answer questions such as:

- "Brief me on this person before a meeting."
- "Give me the context I need for this project."
- "What matters right now for this scope?"
- "What should I know, what is uncertain, and what is missing?"

The goal is to make the open brain useful to a first-party agent without
coupling the agent to engine-specific retrieval details.

---

## Why This Session Exists

The canonical memory foundation is now rich enough to support:

- exact/raw recall
- semantic recall
- graph/timeline reads
- composed retrieval

But first-party agents should not each reinvent:

- what to fetch
- how to merge results
- how to summarize scope/person/project state
- how to expose provenance and uncertainty

Session 9.2 creates that reusable "pre-think" layer.

This also creates the clean boundary between:

- **the brain substrate** and
- **the Chief-of-Staff behavior/orchestration layer**

The Chief of Staff should consume context bundles, not hand-roll its own memory
retrieval policy every time.

---

## THE Brain Laws Check

### Law 1: One Public Face

This session must extend the canonical MCP surface rather than create a
Chief-of-Staff-only private memory API. The context builder is a new canonical
tool/service, not a parallel shadow surface.

### Law 2: Zero-Knowledge

Context bundles may assemble references, summaries, and justified excerpts
through trusted retrieval paths, but they must not create a new denormalized
cache of raw memory content in D1, KV, Analytics Engine, or agent-local
operational stores.

### Law 3: Agent Write Policy

This session is a read-side preparation layer. It must not grant the Chief of
Staff new write or action authority. It prepares context only.

## Behavioral Wiring

### Authorization

| Route / Entry Point | Auth Level | Tenant ID Source | When Checked |
|---------------------|-----------|-----------------|-------------|
| `prepare_context_for_agent` via McpAgent | Existing auth/session path | Existing tenant context | Existing auth middleware + DO session bootstrap |
| Internal context-builder service | Trusted internal caller | Service arg / tenant context | At service boundary |

No new intentionally public HTTP route is added.

### Action Layer

No action tools in this session.

This session may recommend a future action classification shape for the Chief of
Staff, but it must not implement send/schedule/write operations.

### Audit

| Event | Table | Action String | Blocking? |
|-------|-------|-------------|-----------|
| Context bundle requested | existing read telemetry only if already present | metadata-only | Non-blocking |
| Context bundle assembled | existing read telemetry only if already present | metadata-only | Non-blocking |

Do not create a noisy new audit trail that stores the assembled context body.

### Compute Routing

| Operation | Tier | Rationale |
|-----------|------|-----------|
| Context assembly orchestration | Sync Worker / service | MCP request |
| Raw retrieval | Existing canonical read path | Exact provenance/source grounding |
| Semantic retrieval | Existing semantic path | High-level meaning / memory synthesis |
| Graph/timeline retrieval | Existing graph path | relationships / evolution / chronology |
| Bundle synthesis | Sync Worker / service | deterministic bundle assembly |

### Other Behavioral Wiring

- The context builder should consume the 9.1 retrieval router rather than
  bypass it unless an explicit narrow exception is justified.
- Bundles should remain structured and inspectable, not opaque prose blobs.
- Provenance, uncertainty, and suggested follow-up questions should be first-
  class parts of the output.
- The context builder should support multiple bundle intents without becoming a
  giant workflow engine.

## Bundle Types

Define a small first set of bundle intents.

### `person`

Use for:

- meeting prep
- relationship brief
- person-specific memory refresh

Expected content:

- who this is
- recent interactions
- relationship/timeline notes
- open loops / unresolved items
- sources / provenance
- confidence / missing information

### `project`

Use for:

- project refresh
- weekly operating context
- current-state brief

Expected content:

- project summary
- active themes
- recent changes
- risks / blockers
- important people / relationships
- timeline / milestones if known
- sources / provenance

### `scope`

Use for:

- domain or area brief
- a more general operational context bundle

Expected content:

- top facts
- relevant recent memories
- semantic patterns
- graph/timeline highlights when present

### `meeting_prep`

Use for:

- prepare context before a specific meeting or interaction

Expected content:

- person + project blend
- recent interactions
- likely agenda / open questions
- relationship history signals
- unresolved follow-ups

The first rollout may implement `person` and `project` as mandatory, with
`scope` and `meeting_prep` allowed as additive if naturally supported.

## Part 3: Types / Contract

The public contract should define shapes along these lines:

```typescript
export type ContextBundleIntent =
  | 'person'
  | 'project'
  | 'scope'
  | 'meeting_prep'

export interface PrepareContextForAgentInput {
  agent: 'chief_of_staff' | string
  intent: ContextBundleIntent
  target: string
  scope?: string | null
  limit?: number
}

export interface ContextSourceRef {
  mode: 'raw' | 'semantic' | 'graph' | 'composed'
  captureId?: string | null
  documentId?: string | null
  projectionRef?: string | null
}

export interface ContextGap {
  kind: 'missing' | 'uncertain' | 'stale'
  message: string
}

export interface AgentContextBundle {
  intent: ContextBundleIntent
  target: string
  summary: string
  highlights: string[]
  openLoops: string[]
  risks: string[]
  timeline: string[]
  relationships: string[]
  sources: ContextSourceRef[]
  gaps: ContextGap[]
}
```

Exact names may differ, but the contract must answer:

- how does a caller request a context bundle?
- what target is the bundle about?
- what parts of the result are facts vs gaps vs suggestions?
- how does the caller see provenance?

## Part 4: Services

### `src/services/chief-of-staff-context.ts`

Responsibilities:

- accept bundle intent + target
- query the retrieval router and supporting read services
- assemble a deterministic context bundle
- preserve provenance and uncertainty

Responsibilities it must **not** take on:

- taking actions
- mutating source systems
- long-lived scheduling/orchestration
- owning the agent behavior loop

### Retrieval policy

Recommended default policy:

- `person`
  - semantic recall first
  - graph relationship/timeline additions
  - raw confirmation for key recent/source-grounded items

- `project`
  - composed retrieval first
  - semantic recall for themes
  - graph for people/relationship/timeline context
  - raw references for key recent changes

- `scope`
  - composed or semantic-first depending scope size

- `meeting_prep`
  - person + project blend with recency bias

The policy may be simplified in the first rollout, but it must be explicit.

## Part 6: Chief-of-Staff Boundary

This session should define the boundary between:

### Context builder

- "What does the Chief of Staff need to know before acting?"

### Chief-of-Staff agent behavior

- "What should the Chief of Staff do next?"

Only the first belongs here.

This means:

- classification systems such as `dispatch / prep / yours / skip` may inform
  future CoS behavior, but they should not become the main output of 9.2 unless
  justified as metadata only
- no morning sweep or time-blocking workflow is implemented here
- no parallel-subagent orchestration is implemented here

  contract shape, provenance retention, and gap signaling.

  expectation updated for the additive 9.2 tool.

**Migration Decision**
- `1020_chief_of_staff_context_builder.sql` was **not needed**.
- Session 9.2 remained migration-free and reused the existing canonical
  capture/document/projection rows plus the current semantic and graph
  projection readbacks.

**Exact Surface Shipped**
- `prepare_context_for_agent`
- Inputs:
  - `agent`
  - `intent`
  - `target`
  - optional `scope`
  - optional `limit`
- Mandatory bundle support shipped:
  - `person`
  - `project`
- Additive thin-policy support also shipped without widening product scope:
  - `scope`
  - `meeting_prep`
- Output is structured and inspectable:
  - `summary`
  - `confidence`
  - `highlights`
  - `recentChanges`
  - `openLoops`
  - `risks`
  - `timeline`
  - `relationships`
  - `followUpQuestions`
  - `gaps`
  - `sources`
  - `evidence`

**Retrieval Policy Shipped**
- Every bundle starts with one router-driven query so Session 9.2 consumes the
  9.1 multi-mode router rather than bypassing it.
- `person`
  - routed composed seed via `Brief me on {target}`
  - semantic recall via `What do I know about {target}?`
  - graph recall via `How has my relationship with {target} changed over time?`
  - raw/source-grounded lookup via `{target}`
- `project`
  - routed composed seed via `Overview of {target}`
  - semantic recall via `What do I know about {target}?`
  - graph/timeline lookup via `Timeline for {target}`
  - raw/source-grounded lookup via `{target}`
- Bundle synthesis remains deterministic and read-only. No action proposal,
  scheduling, or orchestration behavior was added.

**Provenance / Uncertainty Behavior**
- Source refs preserve canonical `captureId` / `documentId` plus projection
  references where available.
- Public output does **not** expose internal Hindsight engine document or
  operation ids; those stay inside the existing lower-level status surfaces.
- Missing or unavailable retrieval modes become explicit `gaps` rather than
  silent omission.
- Sparse graph coverage degrades gracefully: raw + semantic evidence still
  produce a useful bundle while the missing relationship/timeline signal is
  called out explicitly.

**Tests / Checks Run**
- `npx vitest run tests/9.2-chief-of-staff-context-builder.test.ts`

**Deviations From Planned Scope**
- `scope` and `meeting_prep` were implemented as thin aliases on the same
  read-only policy framework because doing so was a small, architecture-safe
  additive step once the bundle contract existed.
- The first rollout uses structured evidence blocks and source refs rather than
  inventing separate `get_person_context` / `get_project_context` tools. This
  keeps Session 9.2 on the canonical memory surface and defers any later
  first-party naming decisions to future agent-behavior sessions.
