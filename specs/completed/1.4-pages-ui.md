## What We're Building

A Cloudflare Pages React SPA with three surfaces:

1. **Approval Queue** — the main daily-use surface. Shows `awaiting_approval`
   actions with Approve / Reject buttons. Shows `completed_reversible` actions
   with a live countdown undo button. Real-time updates via WebSocket.

2. **Activity Log** — paginated history of all actions across all terminal states
   (`completed`, `undone`, `failed`, `rejected`, `cancelled`, `expired`).

3. **Settings** — authorization preferences per capability class, AI cost
   tracking, tenant contact info. Google connection stub. Bootstrap stub.

When complete: Matt can open the Pages app, see pending agent actions, approve or
reject them, undo a calendar event within 5 minutes, and configure how aggressive
THE Brain is allowed to be with external writes.

## THE Brain Laws Check

### Law 1: One Public Face
Pages app is gated by CF Access — same Access application as the Worker. The Pages
project must be added to the CF Access policy (manual step — document in pre-flight).
Pages Functions are the thin API layer; all business logic stays in the Worker.
Pages Functions proxy to Worker HTTP routes — they do not touch D1, queues, or R2
directly.

**CF Access + Pages:** CF Access service tokens work for Pages projects. The
Pages app at `brain.pages.dev` (or custom domain) gets the same Access policy as
the Worker. Browser sessions with a valid CF Access cookie can call both.

### Law 2: Zero-Knowledge
Pages UI never displays decrypted content. The approval queue shows:
- `tool_name`, `integration`, `capability_class` — metadata only
- `action_type`, `proposed_by`, `proposed_at` — metadata only
- `result_summary` — brief outcome metadata (spec says never content)
- `state`, `authorization_level` — status fields

The actual action payload lives in R2 (encrypted). The UI does not fetch or
display it. If an agent proposes "send SMS to +15551234567 saying X", the UI
shows "send_message via sms" — not the message content.

**This is intentional and correct.** The approval queue is an authorization gate,
not a content viewer.

### Law 3: Agent Write Policy
Not applicable — Pages UI is human-facing only, no agent writes.

## Architecture

```
[Browser] ←→ [Cloudflare Pages (React SPA)]
                      ↓
           [Pages Functions /api/*]
                      ↓ (fetch proxy)
           [McpAgent Worker HTTP routes]
                      ↓
              [D1, R2, KV, DO]
```

**Pages Functions are thin proxies only.** They:
1. Forward the CF Access JWT (already validated by CF before Pages Functions run)
2. Call the Worker's existing HTTP routes
3. Return the response verbatim

No business logic in Pages Functions. No D1 calls from Pages Functions.
This keeps the Worker as the single source of truth for all data operations.

**WebSocket for real-time updates:**
The approval queue subscribes to the McpAgentDO WebSocket (same connection the
MCP client uses). Pages app opens a WebSocket to `GET /mcp` and listens for
`action.*` events. On `action.executed` or `action.proposed`, the queue
re-fetches from the Worker API.

## New Worker Routes (added to McpAgent Worker)

Pages Functions proxy to these. All behind CF Access auth + audit middleware.

| Method | Path | Purpose |
|--------|------|---------|
| `GET` | `/api/actions` | List actions — filterable by state, paginated |
| `POST` | `/api/actions/:id/approve` | Approve awaiting_approval action |
| `POST` | `/api/actions/:id/reject` | Reject awaiting_approval action |
| `POST` | `/actions/:id/undo` | Already exists from 2.3 — no change |
| `GET` | `/api/settings` | Read tenant settings + auth preferences |
| `POST` | `/api/settings/preferences` | Write auth preference (HMAC-protected) |
| `GET` | `/api/audit` | Paginated action_audit log |

**`GET /api/actions` query params:**
```
state=awaiting_approval|completed_reversible|completed|failed|rejected|all
limit=20 (default)
offset=0
```

**`POST /api/actions/:id/approve` behavior:**
```typescript
// 1. Validate action is in awaiting_approval state
// 2. Update state: awaiting_approval → queued
// 3. Set execute_after = now + send_delay_seconds * 1000
// 4. Set approved_by = jwtSub (from CF Access JWT)
// 5. Set approved_at = now
// 6. Atomic db.batch(): state UPDATE + action_audit INSERT (action.approved)
// 7. McpAgentDO.broadcast({ type: 'action.approved', action_id })
// Returns: { action_id, state: 'queued', execute_after }
```

**`POST /api/actions/:id/reject` behavior:**
```typescript
// 1. Validate action is in awaiting_approval state
// 2. Update state: awaiting_approval → rejected
// 3. Set cancel_reason = body.reason (optional)
// 4. Atomic db.batch(): state UPDATE + action_audit INSERT (action.rejected)
// 5. McpAgentDO.broadcast({ type: 'action.rejected', action_id })
// Returns: { action_id, state: 'rejected' }
```

**`POST /api/settings/preferences` behavior:**
```typescript
// Writes to tenant_action_preferences table
// Must call computePreferenceHmac() and store row_hmac
// Input: { capability_class, integration, authorization_level }
// Validates: authorization_level >= hard_floor for capability_class
// Returns: { capability_class, integration, authorization_level, updated_at }
```

## UI Surfaces

### Surface 1: Approval Queue

**Layout:** Three-column card grid on desktop, single-column on mobile.

**Card anatomy** (for each `awaiting_approval` action):
```
┌─────────────────────────────────────────────┐
│ [YELLOW] WRITE_EXTERNAL_REVERSIBLE          │
│ brain_v1_act_create_event · calendar        │
│ Proposed by: chief_of_staff · 3 min ago     │
│                                             │
│ [Approve]              [Reject ▾]           │
└─────────────────────────────────────────────┘
```

**Undo card** (for each `completed_reversible` action):
```
┌─────────────────────────────────────────────┐
│ [GREEN] Executed                            │
│ brain_v1_act_create_event · calendar        │
│ Completed · just now                        │
│                                             │
│ [Undo — 4:32 remaining]                     │
└─────────────────────────────────────────────┘
```

Countdown timer: live, ticks every second from `undo_expires_at - Date.now()`.
When expired: button disappears, card moves to activity log on next refresh.

**Empty state:**
```
┌─────────────────────────────────────────────┐
│         ✓  Nothing pending                  │
│   THE Brain has no actions awaiting         │
│         your approval.                      │
└─────────────────────────────────────────────┘
```

**Reject modal:** On "Reject ▾" click, show inline modal with optional
reason field. "Reject without reason" and "Reject with reason" options.

**Authorization level indicator:**
Each card shows a color-coded badge:
- GREEN — auto-executed (should not appear in approval queue, but shown in activity log)
- YELLOW — awaiting approval (primary queue state)
- RED — blocked (appears in activity log as rejected-by-system)

**Real-time behavior:**
- On mount: fetch `GET /api/actions?state=awaiting_approval` + `state=completed_reversible`
- Subscribe to WebSocket `action.*` events → re-fetch on any event
- Optimistic UI: on Approve click, immediately move card to "queued" visual state
  before server confirmation. Revert on error.

## Part 2: Pages Function Proxy

```typescript
// pages/functions/api/[[catchall]].ts
// Thin proxy — all logic stays in Worker

export async function onRequest(context: EventContext<Env, string, unknown>) {
  const workerUrl = context.env.WORKER_URL  // set in Pages env vars
  const url = new URL(context.request.url)

  // Forward to Worker, preserving path, method, body, CF Access headers
  const workerRequest = new Request(
    `${workerUrl}${url.pathname}${url.search}`,
    {
      method: context.request.method,
      headers: context.request.headers,
      body: context.request.body,
    }
  )

  return fetch(workerRequest)
}
```

**Pages environment variables required:**
- `WORKER_URL` — the Worker's URL (e.g., `https://the-brain.workers.dev`)
  Set in Cloudflare Pages dashboard → Settings → Environment variables.

## Part 3: Key Component Patterns

```typescript
// src/hooks/useCountdown.ts
export function useCountdown(expiresAt: number): { remaining: number; expired: boolean } {
  const [remaining, setRemaining] = useState(() => Math.max(0, expiresAt - Date.now()))

  useEffect(() => {
    if (remaining === 0) return
    const interval = setInterval(() => {
      const r = Math.max(0, expiresAt - Date.now())
      setRemaining(r)
      if (r === 0) clearInterval(interval)
    }, 1000)
    return () => clearInterval(interval)
  }, [expiresAt])

  return { remaining, expired: remaining === 0 }
}

// src/hooks/useActions.ts
export function useActions() {
  const [actions, setActions] = useState<ActionRow[]>([])
  const [loading, setLoading] = useState(true)

  const fetchActions = useCallback(async () => {
    const res = await apiClient.get('/api/actions?state=awaiting_approval,completed_reversible')
    setActions(res.actions)
    setLoading(false)
  }, [])

  useEffect(() => {
    fetchActions()

    // WebSocket subscription for real-time updates
    const ws = new WebSocket(`${WS_URL}/mcp`)
    ws.onmessage = (e) => {
      const msg = JSON.parse(e.data)
      if (msg.type?.startsWith('action.')) fetchActions()
    }
    return () => ws.close()
  }, [fetchActions])

  const approve = useCallback(async (actionId: string) => {
    await apiClient.post(`/api/actions/${actionId}/approve`)
    await fetchActions()
  }, [fetchActions])

  const reject = useCallback(async (actionId: string, reason?: string) => {
    await apiClient.post(`/api/actions/${actionId}/reject`, { reason })
    await fetchActions()
  }, [fetchActions])

  return { actions, loading, approve, reject, refetch: fetchActions }
}

// src/components/UndoCard.tsx — undo countdown
export function UndoCard({ action }: { action: ActionRow }) {
  const { remaining, expired } = useCountdown(action.undo_expires_at)
  const minutes = Math.floor(remaining / 60_000)
  const seconds = Math.floor((remaining % 60_000) / 1000)

  return (
    <div className="action-card completed-reversible">
      <div className="badge green">Executed</div>
      <div className="tool-name">{action.tool_name} · {action.integration}</div>
      <button
        disabled={expired}
        onClick={() => onUndo(action.id)}
        className={expired ? 'btn-disabled' : 'btn-undo'}
      >
        {expired ? 'Undo expired' : `Undo — ${minutes}:${String(seconds).padStart(2, '0')} remaining`}
      </button>
    </div>
  )
}
```

## Part 4: Worker Route — Approve/Reject

```typescript
// Addition to src/workers/mcpagent/routes/actions.ts

actionsRouter.get('/', async (c) => {
  const tenantId = c.get('tenantId')
  const state = c.req.query('state') ?? 'awaiting_approval'
  const limit = parseInt(c.req.query('limit') ?? '20')
  const offset = parseInt(c.req.query('offset') ?? '0')

  const stateFilter = state === 'all'
    ? ''
    : `AND state IN (${state.split(',').map(() => '?').join(',')})`
  const stateValues = state === 'all' ? [] : state.split(',')

  const rows = await c.env.D1_US.prepare(
    `SELECT id, proposed_at, proposed_by, tool_name, capability_class,
            integration, action_type, state, authorization_level,
            send_delay_seconds, execute_after, approved_by, approved_at,
            executed_at, result_summary, episodic_memory_id
     FROM pending_actions
     WHERE tenant_id = ? ${stateFilter}
     ORDER BY proposed_at DESC
     LIMIT ? OFFSET ?`
  ).bind(tenantId, ...stateValues, limit, offset).all()

  return c.json({ actions: rows.results, limit, offset })
})

actionsRouter.post('/:id/approve', async (c) => {
  const tenantId = c.get('tenantId')
  const jwtSub = c.get('jwtSub')
  const actionId = c.req.param('id')
  const now = Date.now()

  // Fetch action — verify state + tenant ownership
  const action = await c.env.D1_US.prepare(
    'SELECT state, send_delay_seconds, capability_class FROM pending_actions WHERE id = ? AND tenant_id = ?'
  ).bind(actionId, tenantId).first<{ state: string; send_delay_seconds: number; capability_class: string }>()

  if (!action) return c.json({ error: 'Not found' }, 404)
  if (action.state !== 'awaiting_approval') return c.json({ error: 'Not awaiting approval' }, 409)

  const executeAfter = now + action.send_delay_seconds * 1000

  await c.env.D1_US.batch([
    c.env.D1_US.prepare(
      `UPDATE pending_actions
       SET state = 'queued', approved_by = ?, approved_at = ?, execute_after = ?
       WHERE id = ?`
    ).bind(jwtSub, now, executeAfter, actionId),
    c.env.D1_US.prepare(
      `INSERT INTO action_audit (id, tenant_id, action_id, created_at, event, agent_identity)
       VALUES (?, ?, ?, ?, 'action.approved', ?)`
    ).bind(crypto.randomUUID(), tenantId, actionId, now, jwtSub),
  ])

  // Broadcast via McpAgentDO
  const id = c.env.MCPAGENT.idFromName(tenantId)
  const stub = c.env.MCPAGENT.get(id) as DurableObjectStub<McpAgentDO>
  // @ts-expect-error DO RPC
  stub.broadcast({ type: 'action.approved', action_id: actionId, tenant_id: tenantId })

  return c.json({ action_id: actionId, state: 'queued', execute_after: executeAfter })
})
```
