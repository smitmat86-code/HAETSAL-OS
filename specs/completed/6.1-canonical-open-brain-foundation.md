---

## What We're Building

Introduce the canonical open-brain foundation beneath the current HAETSAL
memory system. This session creates the first source-of-truth layer for
captures, artifacts, documents, chunks, memory operations, and projection jobs
without replacing the current live Hindsight path yet.

When complete, HAETSAL will have a canonical brain substrate that every future
memory write can enter first, before fan-out into Hindsight, Graphiti, or other
projections.

---

## THE Brain Laws Check

### Law 1: One Public Face

No new public surface may bypass the current McpAgent/Worker entrypoint. Any
new capture API in this session must still live behind the existing Worker
surface and must not expose Postgres, R2, Hindsight, or future graph services
directly.

### Law 2: Zero-Knowledge

This spec introduces canonical raw-memory records and artifact references.
Memory content written to HAETSAL-owned stores must remain encrypted according to
the current tenant-key model. Raw content must not land in D1, KV, or Analytics
Engine. Artifact payloads stored in R2 must follow the existing encrypted
artifact pattern.

### Law 3: Agent Write Policy

This session introduces a canonical capture path, not new domain-agent write
behavior. Domain agents may only write allowed episodic/semantic/world memory
types. Procedural memory remains cron-only. Any path that touches the existing
memory write layer must preserve the validator already in the production path.

### Audit

| Event | Table | Action String | Blocking? |
|-------|-------|-------------|-----------|
| Canonical capture accepted | canonical audit / existing memory audit lane | `memory.capture.accepted` | Atomic with capture write |
| Canonical projection job created | canonical audit / existing memory audit lane | `memory.projection.enqueued` | Atomic with projection job write |
| Canonical capture failure | existing audit lane | `memory.capture.failed` | Atomic when failure occurs after write begins |

All state-mutating operations and security-relevant failures must generate audit
rows atomically with the underlying write batch.

### Compute Routing

| Operation | Tier | Rationale |
|-----------|------|-----------|
| Canonical capture normalization | Sync Worker / service | Must produce accepted capture receipt immediately |
| Projection job creation | Sync Worker / service | Part of capture acceptance contract |
| Hindsight projection fan-out | Queue C3 | Async semantic projection |
| Future Graphiti projection fan-out | Queue C3 | Async graph projection |
| Artifact parsing beyond request budget | Workflow C4 | Long-running normalization if needed later |

This session must not block sync MCP calls on Hindsight or Graphiti work.

### Other Behavioral Wiring

- Canonical capture service must create projection jobs, not call Hindsight
  directly.
- Existing production Hindsight path stays live during this session.
- A feature-flag or explicit internal mode must allow shadow-write behavior
  while current reads remain unchanged.
- All canonical IDs must map cleanly back to tenant scope and source system.

## Part 4: Core Logic

Core service flow:

1. accept canonical capture input
2. validate tenant/scope/source metadata
3. persist canonical capture row
4. persist linked document and chunks
5. persist memory operation row
6. persist projection jobs for Hindsight and Graphiti
7. write audit rows atomically
8. return canonical capture receipt

Required service API:

```typescript
export async function captureCanonicalMemory(
  input: CanonicalCaptureInput,
  env: Env,
  tenantId: string
): Promise<CanonicalCaptureResult>
```

Artifact helper logic should preserve the current encrypted artifact pattern and
store only references in the canonical rows.

## As-Built Record

**Built**
- `src/types/canonical-memory.ts` introduces the canonical capture/artifact/job
  types and result contract.
- `src/services/canonical-memory.ts` adds the direct service-layer
  `captureCanonicalMemory()` API plus `maybeShadowWriteCanonicalCapture()`.
- `src/services/canonical-memory-schema.ts`,
  `src/services/canonical-memory-types.ts`,
  `src/services/canonical-memory-artifacts.ts`, and
  `src/services/canonical-memory-audit.ts` split validation, chunk planning,
  encrypted R2 persistence, and atomic audit batching.
- `migrations/1013_canonical_open_brain_foundation.sql` creates
  `canonical_captures`, `canonical_artifacts`, `canonical_documents`,
  `canonical_chunks`, `canonical_memory_operations`,
  `canonical_projection_jobs`, and `canonical_projection_results`.
- `tests/fixtures/canonical-memory/*.json` adds note, conversation, and
  artifact fixtures.
- `tests/6.1-canonical-open-brain-foundation.test.ts` verifies note capture,
  multi-chunk conversation capture, artifact linkage, tenant isolation,
  zero-knowledge storage, and shadow-write fan-out job creation.
- `src/services/ingestion/retain.ts` now has an off-by-default best-effort
  shadow-write hook guarded by `CANONICAL_MEMORY_SHADOW_WRITES === "true"`.

**Storage Decision**
- Session 6.1 introduces a **bridge layer first**, not direct Neon/Postgres
  writes. Canonical metadata lives in D1 and canonical raw payloads live as
  encrypted R2 objects referenced by D1 rows.
- The schema and service contracts are shaped 1:1 for later Postgres migration,
  but this session does not add a new Worker-to-Neon runtime path or replace
  the current production Hindsight write lane.

**Why This Deviation Was Chosen**
- The architecture target remains canonical Postgres + R2.
- The current HAETSAL shell does not yet expose a proven Worker-side Neon write
  path outside the Hindsight/container runtime.
- Session 6.1 was constrained to reuse the current production shell, avoid
  replacing the live Hindsight path, and make the narrowest decision consistent
  with the architecture docs.

**Explicit Deviations**
- `CanonicalCaptureInput` gained an optional `bodyEncrypted` field in
  implementation. The service requires it for HAETSAL-owned storage so raw
  canonical body content never lands in D1/KV/Analytics.
- Canonical chunk rows store offsets + hashes, not plaintext chunk bodies.
  The encrypted canonical document body in R2 remains the backing source.
- The optional shadow-write integration is **best-effort via `waitUntil`**
  rather than blocking the existing retain path. Production Hindsight behavior
  remains unchanged when the feature flag is off.
- No new ingestion queue message type was added in 6.1; the existing retain
  service seam was reused instead.
