## What We're Building

Define and expose the stable canonical memory interface for the advanced open
brain. This session introduces the public MCP-facing contract that future AI
clients and first-party HAETSAL agents should depend on, without requiring them
to know whether a response comes from raw canonical data, Hindsight, or a later
graph projection.

When complete, HAETSAL will have a clean universal brain interface centered on
canonical memory concepts rather than engine-specific concepts.

---

## THE Brain Laws Check

### Law 1: One Public Face

All new memory tools must be added through the existing McpAgent/Worker public
surface. No new direct HTTP routes to Postgres, R2, Hindsight, or a future
Graphiti service are allowed. Engine-specific memory behavior remains internal.

### Law 2: Zero-Knowledge

This session adds read interfaces over canonical and operational memory state.
Any path that returns raw content must respect existing decryption and
zero-knowledge rules. No memory content may be written to D1, KV, or Analytics
Engine. Memory status and metadata responses may surface operational state, but
not decrypted content unless the current tenant is authorized and the content is
retrieved through the approved decrypt path.

### Law 3: Agent Write Policy

This session is primarily a read/status/interface spec. If it introduces a
canonical capture tool alias, it must preserve the existing allowed write policy
and must not add any path for procedural writes outside the consolidation cron.

### Action Layer (if applicable)

### Audit

| Event | Table | Action String | Blocking? |
|-------|-------|-------------|-----------|
| Canonical memory search executed | memory audit / canonical audit lane | `memory.search.executed` | Atomic with any stateful side effect, otherwise waitUntil metadata-only |
| Canonical recent list viewed | memory audit / canonical audit lane | `memory.recent.viewed` | waitUntil metadata-only |
| Canonical document retrieved | memory audit / canonical audit lane | `memory.document.viewed` | waitUntil metadata-only |
| Memory status checked | memory audit / canonical audit lane | `memory.status.viewed` | waitUntil metadata-only |

This session must not add misleading audit semantics. Interface reads should be
clearly distinguished from writes and from projection completion state.

### Compute Routing

| Operation | Tier | Rationale |
|-----------|------|-----------|
| Canonical search across current foundation | Sync Worker / service | User-facing memory query |
| Recent list lookup | Sync Worker / service | User-facing memory query |
| Document lookup | Sync Worker / service | User-facing memory query |
| Memory operation status lookup | Sync Worker / service | User-facing / operator-facing status check |
| Future semantic or graph routing | Deferred | Not implemented in this session |

This session defines the contract but must not broaden into full multi-engine
retrieval orchestration.

### Other Behavioral Wiring

- New canonical tools should be registered in the MCP surface alongside, not in
  place of, the current production memory tools unless an explicit aliasing
  decision is made in implementation.
- The public contract should use canonical names and avoid engine-specific
  vocabulary where possible.
- The search path may initially operate over canonical D1 metadata / encrypted
  canonical structures from Session 6.1, with clear note in As-Built if semantic
  recall is intentionally deferred.
- Memory status should expose canonical operation state and projection-job state
  without claiming semantic availability unless actually confirmed.

## As-Built Record

**Built**
- `src/types/canonical-memory-query.ts` - canonical query/result contracts for
  search, recent, document, status, and stats reads
- `src/services/canonical-memory-read-model.ts` - tenant-authorized R2 decrypt
  helper plus shared preview/limit utilities
- `src/services/canonical-memory-query.ts`,
  `src/services/canonical-memory-status.ts`,
  `src/services/canonical-memory-stats.ts` - canonical foundation-first read,
  status, and stats services over Session 6.1 D1 + encrypted R2 records
- `src/tools/canonical-memory.ts` - canonical MCP tool registration for
  `capture_memory`, `search_memory`, `get_recent_memories`, `get_document`,
  `memory_status`, and `memory_stats`
- `src/workers/mcpagent/do/McpAgent.ts` - canonical tools registered through the
  existing McpAgent surface; version bumped to `6.2.0`
- `tests/6.2-canonical-mcp-memory-surface.test.ts` plus
  `tests/fixtures/canonical-memory/{note-search-query,recent-query,document-query,status-query}.json`
  - tool-level coverage for canonical search, recent, document, status, stats,
  capture alias presence, and tenant isolation

**Decisions**
- Canonical reads are foundation-first in this session. `search_memory` and
  `get_recent_memories` read canonical D1 metadata and, when a session TMK is
  present, decrypt canonical document bodies from R2 to build previews and
  query matches.
- `memory_status` reports canonical operation and projection-job truth only. It
  does not claim semantic availability beyond the recorded projection state.
- The canonical tool surface is additive. Legacy `brain_v1_*` and
  `memory_*` tools remain in place so the current production Hindsight path is
  untouched during Session 6.2.

**Explicit Deviations**
- `capture_memory` is a canonical-name bridge to the existing retain path, not
  the 6.3 canonical capture-first pipeline. It maps canonical `scope` to the
  current retain `domain` field so production behavior stays unchanged until the
  canonical write fan-out lands.
- The canonical read services take an optional decrypt context for TMK-backed
  document reads. This keeps the public tool contract canonical while allowing
  the approved decrypt path to remain session-key-scoped.
- Search is not semantic/Hindsight-routed yet. It is a canonical bridge-layer
  search over Session 6.1 D1 metadata plus authorized R2 decrypts, which is
  sufficient for the seeded canonical fixtures and stays within Session 6.2.
