## What We're Building

Choose raw, semantic, graph, or composed retrieval mode based on query intent
through a single canonical memory router.

Today the system has the underlying pieces:

- raw / exact canonical retrieval
- semantic retrieval
- graph / timeline retrieval
- narrow graph-backed composed retrieval

Session 9.1 adds the orchestration layer that decides which mode to use and
returns a consistent canonical response with truthful source attribution.

When complete, callers can ask one canonical memory question and the router can
select:

- raw mode
- semantic mode
- graph mode
- composed mode

without forcing the caller to know which engine should answer, unless they
explicitly want to override it.

## THE Brain Laws Check

### Law 1: One Public Face

The router must live behind the existing McpAgent/Worker canonical MCP surface.
This session must not introduce engine-specific public routes or bypass the
canonical interface.

### Law 2: Zero-Knowledge

The router may orchestrate multiple retrieval sources, but it must not create
new raw-content caches in D1, KV, or Analytics Engine. It should compose
existing trusted retrieval paths and canonical metadata only.

### Law 3: Agent Write Policy

This session is read-path orchestration only. It must not widen any write path,
nor should mode routing introduce side-effectful writes beyond existing
non-blocking read telemetry patterns.

### Action Layer (if applicable)

No action tools in this spec.

### Audit

| Event | Table | Action String | Blocking? |
|-------|-------|-------------|-----------|
| Router-mode selection | existing read telemetry only if already present | metadata-only | Non-blocking |
| Explicit mode override | existing read telemetry only if already present | metadata-only | Non-blocking |

Do not invent a new noisy audit stream just to record every mode decision unless
that already fits the current HAETSAL read telemetry pattern.

### Compute Routing

| Operation | Tier | Rationale |
|-----------|------|-----------|
| Query intent typing / routing | Sync Worker / service | MCP read contract |
| Raw retrieval | Existing canonical/raw path | Already implemented |
| Semantic retrieval | Existing Hindsight-backed canonical path | Already implemented |
| Graph retrieval | Existing Graphiti-backed canonical path | Already implemented |
| Composed retrieval | Existing narrow composed path | Already implemented in bounded form |
| Agent-specific context bundles | Deferred to Session 9.2 | Out of scope |

### Other Behavioral Wiring

- The router should honor both:
  - inferred mode from query intent
  - explicit mode override when supplied by the caller
- Consistent source attribution must be returned regardless of selected mode.
- The router should choose the smallest correct mode first:
  - exact/raw for exact lookup requests
  - semantic for concept/fact recall
  - graph for relationship/timeline questions
  - composed for broader context-building style questions
- Session 9.1 should not become a giant ranking/fusion experiment. It is a
  practical router, not a research system.

## Part 3: Types / Contract

The router contract should define shapes along these lines:

```typescript
export type MemoryQueryMode = 'raw' | 'semantic' | 'graph' | 'composed'

export interface CanonicalMemoryRouteDecision {
  mode: MemoryQueryMode
  reason: string
  explicit: boolean
}

export interface CanonicalAttributedResult {
  mode: MemoryQueryMode
  summary: string
  provenance: {
    sourceSystem?: string | null
    captureId?: string | null
    documentId?: string | null
    projectionRef?: string | null
  }
}
```

Exact names may differ, but the contract must answer:

- what mode was chosen?
- was the mode inferred or explicitly requested?
- how is source attribution represented consistently across all modes?

## Part 7: Implementation Notes

- Prefer simple, explicit routing heuristics over overfitted “AI intent”
  detection.
- Keep router decisions explainable for debugging and future tuning.
- Preserve the architecture examples:
  - “What do I know about X?” -> semantic
  - “How has my relationship with X changed over time?” -> graph
  - “Show me exactly what I said last Thursday.” -> raw
  - “Prepare context for my chief-of-staff agent before a meeting.” -> composed
- Do not broaden into:
  - full agent context bundle assembly
  - retrieval weighting experiments
  - broad cross-engine ranking/fusion research

  context helper now supports both explicit `graph` and routed `composed`
  responses without introducing a new public surface.

- `tests/9.1-multi-mode-memory-router.test.ts` - coverage for inferred raw,

- `tests/7.2-semantic-recall-through-canonical-interface.test.ts`,

  canonical `raw` label replacing the old public-facing `lexical` result label.

**Migration Decision**
- `1020_multi_mode_memory_router.sql` was **not needed**.
- Session 9.1 stayed migration-free and reused the existing canonical metadata,
  semantic recall, graph query, and graph-backed composed retrieval paths.

**Exact Routing Rules Shipped**
- Explicit `mode` always wins. `lexical` is accepted as a backward-compatible
  alias and normalizes to `raw`.
- Inferred mode order:
  - composed for broad context-building phrasing such as "prepare context",
    "brief me on", "what should I know about", or "overview of"
  - graph for relationship/timeline phrasing such as "relationship",
    "timeline", "over time", "how has", or "trace"
  - raw for exact/source-oriented phrasing such as "exactly", "verbatim",
    "what did I say", or document/source/date-like lookups
  - semantic for concept/fact phrasing such as "what do I know about",
    "tell me about", "remember", "recall", or general `who/what/when/where/why/how`
    questions not already claimed by the smaller modes above
  - raw fallback for plain keyword-style queries so existing exact lookup
    behavior remains stable when no stronger intent signal exists
- The router also strips natural-language scaffolding down to a dispatch focus
  term for reuse of the existing narrow graph/composed path:
  - `How has my relationship with User changed over time?` -> `User`
  - `Prepare context for User before a meeting` -> `User`

**Exact Source Attribution Shape Shipped**
- `search_memory` now returns top-level `route` metadata with:
  - `mode`
  - `reason`
  - `explicit`
  - `dispatchQuery`
- Every returned item now includes `attribution` with a consistent shape across
  all modes:
  - `mode`
  - `sourceSystem`
  - `captureId`
  - `documentId`
  - `canonicalOperationId`
  - `projectionKind`
  - `projectionRef`
  - `targetRef`
  - `graphRef`
- Existing mode-specific fields remain intact (`provenance`, `semanticStatus`,
  `graphContext`) so earlier 7.2/8.3 consumers continue to work.

**Deviations From Planned Scope**
- The public canonical result label now uses `raw` rather than `lexical`.

- Routed `composed` mode intentionally reuses the bounded 8.3 graph-backed

  Session 9.2 remains the place for chief-of-staff-style context assembly.
