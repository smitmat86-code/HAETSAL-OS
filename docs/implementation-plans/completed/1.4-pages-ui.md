# Spec 1.4 — Pages UI: Approval Queue + Settings

A Cloudflare Pages React SPA with three surfaces (Approval Queue, Activity Log, Settings) plus the Worker-side API routes that power them. Pages Functions proxy all requests to the existing McpAgent Worker — no D1/R2/queue access from Pages.

## User Review Required

> [!IMPORTANT]
> **Manual pre-step:** The Pages project must be added to the CF Access application (same policy as the Worker). Without this, the Pages app is public.

> [!IMPORTANT]
> **`WORKER_URL` env var** must be set in Cloudflare Pages dashboard → Settings → Environment variables.

> [!WARNING]
> **MANIFEST.md shows `McpAgent DO` and `Cloudflare Pages` as 🔴 Not Built**, but specs 1.2 and 1.3 are already complete based on the test files present. The manifest status table is stale. I will update it during implementation.

> [!NOTE]
> **`undo_expires_at` derivation:** The spec references `undo_expires_at` as a field, but the as-built `pending_actions` table stores `executed_at` only. `UNDO_WINDOW_MS` (300,000 ms / 5 min) is defined in `src/types/action.ts:59`. The `useActions` hook will compute `expiresAt = executed_at + UNDO_WINDOW_MS` before passing to `UndoCard`/`useCountdown()`. Since the Pages SPA is a separate package and can't import from `src/`, a mirrored `UNDO_WINDOW_MS` constant in `pages/src/types/ui.ts` will reference the canonical source in a comment.

## Verification Plan

### Automated Tests

All tests run via:
```
npm test
```
This executes `vitest run` against the `wrangler.test.toml` config using the `@cloudflare/vitest-pool-workers` pool with real D1 bindings.

> [!IMPORTANT]
> **Test approach: `SELF.fetch()` HTTP tests, not direct service function imports.** The 1.3 tests import service functions directly because the `agents` SDK bundling issue prevented `SELF.fetch()` for routes that depend on the McpAgent DO. The 1.4 routes (`approval.ts`, `settings.ts`, `audit.ts`) are plain Hono routes with no `agents` SDK dependency — they can and should be tested through the full middleware chain via `SELF.fetch()`. This requires the new routes to be registered in `test-entry.ts`.
>
> Pattern follows `2.1-sms.test.ts`: `import { SELF } from 'cloudflare:test'` → `SELF.fetch('http://localhost/api/actions', { ... })`.

#### [NEW] `tests/1.4-approval-queue.test.ts` (~120 lines)

HTTP route tests via `SELF.fetch()` against real D1:
1. **GET /api/actions** — returns `awaiting_approval` actions for tenant
2. **GET /api/actions?state=all** — returns all states
3. **POST /api/actions/:id/approve** — transitions to `queued`, sets `approved_by` + `approved_at`
4. **POST /api/actions/:id/approve on non-pending** — returns 409
5. **POST /api/actions/:id/approve wrong tenant** — returns 404
6. **POST /api/actions/:id/reject** — transitions to `rejected`, optional reason stored
7. **POST /api/actions/:id/reject on non-pending** — returns 409

Each test self-contained with its own `ensureTestTenant()` + seed data.

#### [NEW] `tests/1.4-settings.test.ts` (~90 lines)

HTTP route tests via `SELF.fetch()`:
1. **GET /api/settings** — returns auth preferences + tenant settings
2. **POST /api/settings/preferences** — YELLOW → RED accepted (raising)
3. **POST /api/settings/preferences** — RED → YELLOW rejected (lowering below floor)
4. **POST /api/settings/preferences** — HMAC computed server-side (row_hmac set and valid)
5. **POST /api/settings/preferences** — wrong tenant cannot write another's prefs
