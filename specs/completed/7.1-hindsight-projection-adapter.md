## What We're Building

Replace the temporary compatibility retain bridge with the real Hindsight
projection adapter behind the canonical pipeline.

Session 6.3 made canonical capture the first write boundary, but it still
preserved production behavior by calling the old retain path directly through an
explicit compatibility lane. Session 7.1 removes that temporary bridge and
connects canonical projection jobs of kind `hindsight` to a dedicated adapter
worker that:

- reads canonical capture/materialized payload references
- constructs the correct Hindsight retain request
- submits it through the existing HAETSAL Hindsight runtime
- records the Hindsight operation/document/bank references back onto the
  canonical projection result surface
- reconciles async completion honestly

When complete, Hindsight will no longer be the write front door even in a
compatibility sense. It will become an engine-specific semantic projection fed
by canonical capture truth.

## THE Brain Laws Check

### Law 1: One Public Face

All behavior remains behind the current McpAgent/Worker public surface. This
session may change internal queue consumers, projection services, and Hindsight
worker orchestration, but it must not add any new public route that talks
directly to Hindsight, D1, R2, or canonical projection tables.

### Law 2: Zero-Knowledge

Canonical memory content remains encrypted at rest in HAETSAL-owned
infrastructure. Queue payloads, D1 projection jobs, and audit rows must not
contain raw memory bodies. The Hindsight adapter may decrypt/read canonical
content only inside the trusted worker path long enough to call the official
Hindsight API, consistent with the existing production Hindsight posture.

### Law 3: Agent Write Policy

This session changes projection plumbing, not write policy. Agent/user write
validation remains enforced before canonical capture is accepted. The adapter
must not create any new procedural-write path, nor may it bypass the existing
retain policy checks by directly manufacturing Hindsight writes outside the
canonical pipeline.

## Behavioral Wiring

### Authorization

| Route / Entry Point | Auth Level | Tenant ID Source | When Checked |
|---------------------|-----------|-----------------|-------------|
| `capture_memory` / `memory_write` via McpAgent | CF Access session | Existing McpAgent tenant context | Existing auth middleware + DO session bootstrap |
| Canonical projection dispatch queue consumer | Internal only | trusted queue payload `tenantId` | Before adapter execution |
| Hindsight status reconciliation / poll path | Internal only | canonical operation + trusted tenant context | Before status update |

No intentionally public route is added in this spec.

### Action Layer (if applicable)

No action tools in this spec.

### Audit

| Event | Table | Action String | Blocking? |
|-------|-------|-------------|-----------|
| Hindsight projection submission started | memory audit / canonical audit lane | `memory.projection.hindsight_started` | Atomic with projection result/state write |
| Hindsight projection queued in engine | memory audit / canonical audit lane | `memory.projection.hindsight_queued` | Atomic with canonical reconciliation write |
| Hindsight projection became available/completed | memory audit / canonical audit lane | `memory.projection.hindsight_completed` | Atomic with canonical result/state write |
| Hindsight projection failed | memory audit / canonical audit lane | `memory.projection.hindsight_failed` | Atomic with canonical result/state write |

Audit must describe real adapter lifecycle. Do not emit `completed` when only
Hindsight accepted async work, and do not imply recall/read-path migration in
this session.

### Compute Routing

| Operation | Tier | Rationale |
|-----------|------|-----------|
| Canonical capture acceptance | Sync Worker / service | Already delivered in 6.3 |
| Projection dispatch enqueue | Queue C3 | Durable fan-out boundary |
| Hindsight adapter submission | Queue consumer / service | Engine projection should not block MCP response |
| Hindsight async reconciliation | Existing async/poller path reused through canonical reconciliation service | Preserve truthful operation lifecycle |
| Canonical recall surface replacement | Deferred to Session 7.2 | Out of scope |

### Other Behavioral Wiring

- Canonical queue payloads continue to carry identifiers only, never raw memory
  content.
- The Hindsight adapter should reuse the current HAETSAL shell wherever
  possible:
  - existing Hindsight transport/runtime
  - current bank bootstrap/config path
  - current operation status polling semantics
  - current audit and D1 batch conventions
- Canonical projection jobs of kind `hindsight` should become the sole driver
  of Hindsight retain work. The direct compatibility bridge introduced in 6.3
  should be removed or reduced to a transitional no-op inside this session.
- Canonical projection results must store enough engine reference material to
  support later semantic reads without inventing a second source of truth:
  `bank_id`, `document_id`, `operation_id`, and any stable target reference
  needed by later Session 7.2 reads.

## Part 4: Services

### `src/services/canonical-hindsight-projection.ts`

Responsibilities:

- load canonical capture/materialized payload references for a projection job
- recover the content needed for Hindsight submission from canonical storage
- build the official Hindsight retain request using the existing transport
  helpers/patterns
- submit retain through the existing Hindsight runtime
- write canonical projection result/state updates atomically
- emit truthful audit rows

Constraints:

- do not duplicate canonical capture logic
- do not duplicate Hindsight auth/bootstrap logic if existing services already
  provide it
- if Hindsight returns async operation ids, record them for reconciliation

### `src/services/canonical-hindsight-reconcile.ts`

Responsibilities:

- reconcile Hindsight async status back onto canonical projection state
- map engine-specific statuses into canonical `queued/completed/failed`
- update `target_ref`, error message, and completion timestamps truthfully
- support the later Session 7.2 read surface by preserving stable references

If current Hindsight polling utilities already exist, wrap/reuse them rather
than rebuilding a separate poller.

### `src/services/canonical-capture-compat.ts`

Responsibilities:

- remove or retire the direct compatibility retain bridge
- if a thin transition shim remains, it should delegate immediately into the
  real Hindsight projection adapter rather than perform direct retain work on
  its own

The As-Built section must clearly state whether this file was removed, reduced
to a thin adapter, or retained for interface compatibility only.

## Part 5: Queue / Worker Execution

The existing canonical projection consumer should now fan out `hindsight`
projection jobs into the real adapter.

Expected flow:

1. `capture_memory` / `memory_write`
2. canonical capture accepted
3. projection job created as `accepted`
4. dispatch message enqueued
5. queue consumer receives `canonical_projection_dispatch`
6. Hindsight adapter submits retain
7. canonical projection result/job updated to `queued` or `completed`
8. async reconciliation updates canonical state to `completed` or `failed`

Queue payloads must remain metadata-only. If the adapter needs the body, it must
load it from canonical storage during execution.

**Migration Decision**
- `1014_hindsight_projection_adapter.sql` **was required**.
- Session 6.1/6.3 schema was sufficient for status transitions, but it was not
  sufficient for truthful Hindsight reference storage or canonical-to-Hindsight
  operation reconciliation. The additive engine-reference columns were the
  smallest truthful schema extension.

**Compatibility Bridge**
- The temporary direct retain bridge was **retired as a writer**.
- `canonical-capture-compat.ts` now returns a queued compatibility alias only;
  it no longer calls Hindsight, writes `hindsight_operations`, or drives the
  engine lifecycle itself.
- `src/services/canonical-capture-compat-state.ts` was removed because the real
  Hindsight projection result surface now backs both canonical status and the
  legacy compatibility alias.

**Canonical <-> Hindsight Mapping**
- Canonical capture -> Hindsight submission is driven by the canonical
  `hindsight` projection job id.
- Canonical capture body remains TMK-encrypted in canonical R2 at
  `canonical/{tenant}/documents/{documentId}.enc`.
- A Hindsight-only projection payload is materialized at
  `canonical/{tenant}/projections/hindsight/{captureId}.enc`, encrypted with
  the existing tenant KEK so the trusted worker path can decrypt it without
  putting raw content in D1, KV, Analytics Engine, or queue payloads.
- Expected Hindsight document ids are deterministic:
  `buildHindsightDocumentId(tenantId, sourceSystem, sourceRef || captureId)`.
- `canonical_projection_results` now stores:
  `engine_bank_id` = resolved Hindsight bank id,
  `engine_document_id` = deterministic Hindsight document id,
  `engine_operation_id` = async Hindsight operation id when present,
  `target_ref` = `hindsight://banks/{bank}/documents/{document}` with
  `/operations/{operation}` appended for async submissions.
- `hindsight_operations.source_document_id` remains the Hindsight document id,
  which is what the poll/reconcile path uses to mirror completion/failure back
  onto the canonical projection surface.

**Explicit Deviations**
- The spec sketched a possible `canonical-memory-status.ts` modification and a
  two-file adapter/reconcile shape. The implementation needed two additional
  small helper modules (`...payload.ts` and `...state.ts`) solely to satisfy the
  repo’s actual global 150-line postflight limit.
- Instead of storing a D1 pointer to the Hindsight-only payload, the adapter
  uses a deterministic R2 key derived from `tenantId + captureId`. This avoided
  a second schema change while keeping queue payloads metadata-only.
- The stable public response shape from 6.3 is preserved, but the
  `compatibility_status` / `compatibility` fields are now aliases over the real
  canonical Hindsight projection lane rather than over a separate
  `compatibility_*` temporary state machine.
- `retainContent()` now returns the predicted Hindsight document id immediately
  as the stable `memoryId`/`memory_id` surface while the actual async Hindsight
  operation id is recorded later on the canonical projection result row.
