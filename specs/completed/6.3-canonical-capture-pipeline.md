## What We're Building

Make the canonical event pipeline the first write step for live memory capture.
This session rewires the existing memory write entrypoints so accepted writes
land in the canonical foundation first, then publish queue work for future
projection fan-out, while preserving the current production Hindsight behavior
through an explicit compatibility lane.

When complete, HAETSAL will stop treating direct Hindsight retain as the write
front door. Live writes will produce canonical capture truth, projection jobs,
and queue messages first, creating the migration seam for Session 7.x without
breaking current production recall behavior.

---

## THE Brain Laws Check

### Law 1: One Public Face

All canonical-first write behavior must remain behind the existing
McpAgent/Worker public surface. This session may rewire the implementation
behind `capture_memory`, `memory_write`, and the current retain service, but it
must not introduce a new direct HTTP surface to D1, R2, Hindsight, or a future
Graphiti worker.

### Law 2: Zero-Knowledge

This session handles live memory writes and artifact normalization. Canonical
payloads stored in HAETSAL-owned infrastructure must be encrypted before they
are written to R2. No raw memory content may be written to D1, KV, or Analytics
Engine. If compatibility-mode Hindsight retain remains active, it may continue
to send plaintext through the official Hindsight API exactly as the existing
production path already allows, but canonical storage must still remain
encrypted.

### Law 3: Agent Write Policy

This session rewires write plumbing, not write-policy rules. Any path that
accepts agent or user memory writes must preserve the current validator and must
not introduce a path for `memory_type = procedural` outside the consolidation
cron. `capture_memory` and any compatibility retain bridge must continue to
respect the existing allowed memory types.

## Behavioral Wiring

### Authorization

| Route / Entry Point | Auth Level | Tenant ID Source | When Checked |
|---------------------|-----------|-----------------|-------------|
| `capture_memory` via McpAgent tools | CF Access session | McpAgent tenant context / DO props | Existing auth middleware + McpAgent session bootstrap |
| Existing retain-compatible tool/service entrypoints | CF Access session or trusted internal caller | McpAgent tenant context / trusted arg | Existing auth middleware or service boundary |
| Queue consumer handling canonical fan-out dispatch | Internal only | queue payload `tenantId` from trusted producer | Before dispatch processing |

No intentionally public route is added in this spec.

### Action Layer (if applicable)

No action tools in this spec.

### Audit

| Event | Table | Action String | Blocking? |
|-------|-------|-------------|-----------|
| Canonical pipeline capture accepted | memory audit / canonical audit lane | `memory.capture.accepted` | Atomic with capture write |
| Projection job dispatch queued | memory audit / canonical audit lane | `memory.projection.queued` | Atomic with projection-job state update |
| Compatibility retain bridge invoked | memory audit / canonical audit lane | `memory.capture.compatibility_retain_started` | Atomic with compatibility state write or waitUntil metadata-only if no state change |
| Canonical pipeline failure after write begins | memory audit / canonical audit lane | `memory.capture.failed` | Atomic with failure-state write |

Audit records must remain truthful about whether work is accepted, queued,
compatibility-bridged, or failed. This session must not imply that Hindsight or
Graphiti projection completed when only queue handoff occurred.

### Compute Routing

| Operation | Tier | Rationale |
|-----------|------|-----------|
| Canonical capture persistence + projection job creation | Sync Worker / service | Capture acceptance contract |
| Projection queue message publish | Queue C3 | Fire-and-forget fan-out handoff |
| Compatibility retain bridge to current production path | Sync Worker / service with existing async retain semantics | Preserve production behavior while canonical-first lane is introduced |
| Future Hindsight projection consumer | Deferred to Session 7.1 | Not implemented in this session |
| Future Graphiti projection consumer | Deferred to Session 8.x | Not implemented in this session |

This session must not block sync MCP calls on projection execution. Queue handoff
is the durable boundary.

### Other Behavioral Wiring

- The canonical pipeline should reuse the current HAETSAL shell:
  McpAgent tools, existing retain service seams, existing queue bindings, and
  current audit/ops patterns.
- A queue payload schema must be defined for canonical projection dispatch, with
  enough information to identify tenant, capture, operation, and projection
  kinds without embedding raw memory content.
- Compatibility mode must be explicit and documented in As-Built. The intended
  shape is canonical-first acceptance plus current-production retain bridge until
  Session 7.1 replaces the bridge with a real projection adapter.
- If a feature flag is needed to control rollout from shadow mode to
  canonical-first mode, it must be narrow, internal, and default-safe.

## Part 4: Core Logic

This session introduces the canonical-first write orchestrator:

1. validate live capture input and existing write-policy constraints
2. persist canonical capture/document/chunk/artifact/operation rows
3. create projection jobs
4. enqueue canonical projection-dispatch work
5. optionally invoke the current-production compatibility retain bridge
6. return a truthful capture receipt

Required service signatures:

```typescript
export async function captureThroughCanonicalPipeline(
  input: CanonicalPipelineCaptureInput,
  env: Env,
  tenantId: string,
  ctx?: Pick<ExecutionContext, 'waitUntil'>
): Promise<CanonicalCapturePipelineResult>

export async function enqueueCanonicalProjectionDispatch(
  message: CanonicalProjectionDispatchMessage,
  env: Env,
  ctx?: Pick<ExecutionContext, 'waitUntil'>
): Promise<void>

export async function runCompatibilityRetainBridge(
  input: CanonicalPipelineCaptureInput,
  env: Env,
  tenantId: string,
  ctx?: Pick<ExecutionContext, 'waitUntil'>
): Promise<CompatibilityRetainResult>
```

Implementation notes:

- canonical acceptance must happen before compatibility retain starts
- compatibility retain is a migration bridge, not the new source of truth
- dispatch enqueue must be durable and must not include plaintext body content
- queue consumers introduced in this session should only perform dispatch
  bookkeeping or prepare for later projection workers, not implement Hindsight
  projection itself

  projection-dispatch, and compatibility-bridge contracts for Session 6.3.

  entrypoint.

  `capture_memory` surface keep their stable names while returning the current
  Hindsight-visible result plus canonical ids/status metadata.

  `canonical-memory-status.ts`, `canonical-memory-stats.ts`, and
  `src/workers/ingestion/consumer.ts` were updated so accepted/queued/failed and
  compatibility-bridge states stay truthful through the existing HAETSAL shell.
- `tests/6.3-canonical-capture-pipeline.test.ts` verifies canonical-first note,
  conversation, and artifact capture, metadata-only queue payloads, compatibility
  bridging, and procedural-write rejection.

**Storage / State Decision**
- Session 6.3 did **not** require `1014_canonical_capture_pipeline.sql`.
- The existing Session 6.1 tables were sufficient once the implementation
  reused `canonical_projection_jobs.status` for accepted/queued/failed and
  reused `canonical_projection_results` for queue-handoff plus compatibility
  bridge status.

**Explicit Deviations**
- The queue message shape is wrapped in the repo’s existing queue envelope
  (`type`, `tenantId`, `payload`, `enqueuedAt`) rather than sending the flat
  shape shown in Part 3. This keeps Session 6.3 inside the current HAETSAL
  queue shell without introducing a second queue contract style.
- Compatibility bridge state is persisted as `canonical_projection_results`
  rows with `compatibility_*` statuses instead of adding a new table or new
  columns. This avoided a schema change while still making compatibility state
  queryable and truthful.
- `src/services/canonical-capture-compat-state.ts` was added beyond the planned
  file list so the compatibility bridge stays under the repo’s real global
  postflight line limit.
- `enqueue.ts` did not need a separate edit once the canonical dispatch handoff
  lived in `canonical-projection-dispatch.ts`; the existing queue shell was
  still reused through `QUEUE_BULK`.
