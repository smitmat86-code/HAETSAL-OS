## What We're Building

Expose relationship and time-aware memory reads through the canonical MCP
surface.

Session 8.2 made Graphiti a live canonical projection target. Session 8.3 turns
that projection into a user-facing canonical query surface by adding graph-mode
reads such as:

- `trace_relationship`
- `get_entity_timeline`
- graph-backed composed retrieval

When complete, clients can ask for graph and timeline reasoning through the
canonical interface without talking about Graphiti directly unless they are
explicitly asking for engine distinctions.

---

## THE Brain Laws Check

### Law 1: One Public Face

All graph/timeline query behavior must stay behind the current McpAgent/Worker
surface. This session may extend canonical tools and internal services, but it
must not introduce a direct Graphiti API route.

### Law 2: Zero-Knowledge

Graph/timeline queries must continue to rely on canonical metadata plus trusted
projection/runtime paths. No raw memory content may be copied into D1, KV, or
Analytics Engine just to make graph queries easier.

### Law 3: Agent Write Policy

This session is read-only from a product perspective. It must not create a new
write path or broaden write permissions while adding graph/timeline retrieval.

## Part 3: Types / Contract

The public canonical graph query contract should define shapes along these
lines:

```typescript
export interface TraceRelationshipInput {
  from: string
  to?: string | null
  relation?: string | null
  limit?: number
}

export interface EntityTimelineInput {
  entity: string
  startAt?: number | null
  endAt?: number | null
  limit?: number
}

export interface GraphContextResult {
  summary: string
  provenance: {
    captureId?: string | null
    documentId?: string | null
    graphRef?: string | null
  }
}
```

Exact names may differ, but the contract must answer:

- how does a caller ask for relationship tracing?
- how does a caller ask for entity evolution over time?
- how does graph-backed composed retrieval come back through the canonical
  surface?

### Exact Graph / Timeline Surface Shipped

- `trace_relationship`
  - explicit canonical MCP tool for direct graph relationship tracing
  - inputs: `from`, optional `to`, optional `relation`, optional `limit`
- `get_entity_timeline`
  - explicit canonical MCP tool for ordered graph-backed entity observations
  - inputs: `entity`, optional `start_at`, optional `end_at`, optional `limit`
- `search_memory`
  - extended with `mode: 'graph'` as the narrow explicit graph-backed composed
    retrieval path
  - existing default lexical mode and semantic mode remain unchanged unless
    explicitly requested

### Graph-Backed Composed Retrieval Scope

- The shipped composed retrieval path is intentionally narrow:
  `search_memory(mode = 'graph')` treats `query` as an explicit entity/topic/
  scope lookup term and returns graph-backed context items with canonical
  provenance.
- It does **not** attempt automatic mode selection, cross-engine ranking, or
  the broader Phase 9 multi-mode router.

### Provenance / Linkback Behavior

- Graph results resolve through `canonical_graph_identity_mappings` plus the
  latest completed `graphiti` projection job/result for each canonical capture.
- Returned graph/timeline items include canonical capture/document/operation
  linkback, projection job/result ids, Graphiti refs, and target refs when
  available.
- No raw memory content is copied into D1, KV, Analytics Engine, or query-side
  caches for graph reads.

- `npm test` - passed

### Deviations From Planned Scope

- `trace_relationship` ships as a direct canonical single-hop relationship
  trace over completed graph projections. It does not attempt arbitrary
  multi-hop traversal.
- The graph query path currently reconstructs canonical graph observations from
  the existing Graphiti projection mappings and canonical metadata rather than
  introducing a new public Graphiti query namespace or a Phase 9-style router.
