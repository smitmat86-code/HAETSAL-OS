# Implementation Plan: Phase 2.4 — Bootstrap Import

## Goal

Transform an empty Brain into a populated one. A durable Cloudflare Workflow orchestrates: **Phase A** (Chief of Staff interview — domain-by-domain Q&A retained as semantic memories), **Phase B** (historical Gmail 12mo + Calendar 24mo + Drive 3yr import via QUEUE_BULK), and **Phase C** (handoff — status update + WebSocket push). All content flows through `retainContent()` from 2.1.

## User Review Required

> [!IMPORTANT]
> **Cloudflare Workflows API.** This is the first use of Cloudflare Workflows in the project. Before implementation, verify:
> 1. `[[workflows]]` binding syntax in wrangler.toml (read current docs)
> 2. `WorkflowEntrypoint` class and `step.do()` semantics
> 3. How Workflows interact with Durable Objects (can a Workflow step call a DO RPC?)
> 4. Local dev behavior with `wrangler dev` — does the Workflow run locally?

> [!NOTE]
> **Interview state persistence — decided.** `InterviewSession` state (domainIndex, questionIndex, accumulated answers) is persisted in McpAgentDO's SQLite storage via `this.ctx.storage.sql`. This survives MCP session drops — the user reconnects and resumes from where they left off. This follows the same pattern the DO already uses for session state. No D1 or in-memory-only alternatives.

> [!NOTE]
> **QUEUE_BULK 128KB — non-issue.** Historical import queue messages contain thread/event IDs and metadata, not content. The consumer fetches content via the Google API at processing time. Message payloads are well under 128KB. No R2 indirection needed.

### Workflow

#### [NEW] [bootstrap.ts](file:///c:/Users/matth/Documents/HAETSAL%20OS/src/workflows/bootstrap.ts)
- `BootstrapWorkflow extends WorkflowEntrypoint<Env, BootstrapParams>`
- Three durable steps via `step.do()`:
  - `await-interview` — polls D1 for `interview_completed_at` (sleeps between polls)
  - `import-gmail` — calls `importGmailHistory()` (12 months)
  - `import-calendar` — calls `importCalendarHistory()` (24 months)
  - `import-drive` — calls `importDriveHistory()` (36 months, capped at 500 docs)
  - `bootstrap-complete` — update tenants status, write summary memory, WS push
- Each `step.do()` is independently retryable and durable across Worker restarts
- ~150 lines

### Bootstrap Services

#### [NEW] [interview.ts](file:///c:/Users/matth/Documents/HAETSAL%20OS/src/services/bootstrap/interview.ts)
- `InterviewSession` class — manages domain-by-domain question flow
- `INTERVIEW_DOMAINS` — 5 domains (identity, career, health, relationships, finance)
  - 2–3 questions each, ~12 total
- `currentQuestion()` → next question or null (complete)
- `recordAnswer(answer, tenantId, ..., env)`:
  - Retains answer via `retainContent()` as `memory_type: 'semantic'`, `provenance: 'user_authored'`, `salience_tier: 3`
  - Advances to next question/domain
  - Returns `{ nextQuestion, complete }`
- ~100 lines

#### [NEW] [historical-import.ts](file:///c:/Users/matth/Documents/HAETSAL%20OS/src/services/bootstrap/historical-import.ts)
- `importGmailHistory(tenantId, ..., monthsBack, env)` — list threads → batch to QUEUE_BULK
- `importCalendarHistory(tenantId, ..., monthsBack, env)` — list events → batch to QUEUE_BULK
- `importDriveHistory(tenantId, ..., monthsBack, env)` — list files → batch to QUEUE_BULK
- `historicalSalienceMultiplier(occurredAt, maxMonths)` — date-weighted decay (min 0.5)
- Batches in groups of 50 to QUEUE_BULK via `sendBatch()`
- Uses Gmail/Calendar/Drive services from 2.2
- **Critical:** QUEUE_BULK only — never QUEUE_HIGH or QUEUE_NORMAL
- ~120 lines

### Queue Consumer Extension

The QUEUE_BULK consumer (from 2.1) needs new message type handlers:

| Message Type | Source | Handler |
|---|---|---|
| `bootstrap_gmail_thread` | Historical import | Fetch thread → retainContent() with `provenance: 'bootstrap_import'` |
| `bootstrap_calendar_event` | Historical import | Fetch event → retainContent() with `provenance: 'bootstrap_import'` |
| `bootstrap_drive_file` | Historical import | Download file → retainContent() with `provenance: 'bootstrap_import'` |

Each handler applies the date-weighted salience multiplier and uses `INSERT OR IGNORE` for idempotency.
