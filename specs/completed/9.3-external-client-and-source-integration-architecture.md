## What We're Building

Define the integration architecture that lets HAETSAL's open brain connect to:

- external AI clients such as Claude Code, Codex, Cursor, and other MCP-native
  tools
- non-MCP web clients such as ChatGPT web, Claude web, and Gemini web through
  a portability bridge
- source systems such as Gmail, Calendar, Drive/Docs, and future project
  systems

This session is not "connect everything at once." It is the architectural and
contract session that decides:

- what the stable external brain surfaces are
- how capability scoping works
- what belongs in memory-only access vs source-read vs source-write
- how Bring Your Own Context (BYOC) portability fits into HAETSAL
- how Google and future source ingestion should work without naive mirroring

The target outcome is a structured, implementation-ready model for getting
information into the brain sooner while staying aligned with:

- HAETSAL's canonical open-brain architecture
- Nate Jones / OB1's "one brain, portable context, scoped access" posture
- current Cloudflare MCP direction around remote MCP, OAuth-ready auth posture,
  and narrower tool surfaces

This is a parallel architecture lane that should inform future implementation
sessions without blocking `9.2 Chief-of-Staff Context Builder`.

## Why This Session Exists

The open-brain foundation is now far enough along that two needs become urgent:

1. **Real ingestion should begin sooner**
   - project work across Codex / Claude Code / other coding agents
   - Google Suite signals
   - future source connectors

2. **The connection pattern should firm up before ad hoc integrations sprawl**
   - not every client should invent its own memory strategy
   - not every source connector should invent its own capture rules
   - not every external access surface should receive the same permissions

OB1's BYOC framing is useful here:

- the brain is one durable substrate
- portable working identity is a first-class artifact
- remote MCP is the interoperability layer
- integrations and recipes sit around the brain instead of replacing it

Cloudflare's latest posture sharpens the delivery side:

- remote MCP over modern transport
- OAuth-native / managed auth direction
- scoped MCP surfaces rather than one giant over-privileged tool catalog
- portals / Code mode as possible delivery shapes when the tool surface grows

This session turns those ideas into a HAETSAL-specific architecture decision.

## THE Brain Laws Check

### Law 1: One Public Face

HAETSAL remains one brain and one canonical substrate. This session may define
multiple scoped integration surfaces, but they must still compose into one
coherent public brain story rather than separate brains per client or source.

### Law 2: Zero-Knowledge

External client and source integration must not justify copying raw email,
documents, or memory content into D1, KV, Analytics Engine, or integration-side
caches. Canonical capture plus R2/Postgres remain the durable truth.

### Law 3: Agent Write Policy

This session must keep write boundaries explicit:

- memory capture/write
- source-read ingestion
- source-write/action

No external integration should silently gain source-write authority just because
it can read or write memory.

---

## Strategic Answer

### One brain or many?

One brain.

### One MCP surface or many?

One conceptual brain interface, implemented as a **small set of scoped
capability surfaces** rather than one universal omnipotent surface.

### Different MCP per vendor?

No.

Claude Code, Codex, Cursor, and similar clients should generally connect to the
same **memory-oriented** brain surface when they need the same capabilities.
Surface differences should be driven by capability scope, not AI brand.

### Mirror sources or selectively ingest?

Selective ingestion with provenance, plus bounded historical import where
needed. Do not build a naive full-copy mirror of every source system.

### Should BYOC be part of HAETSAL?

Yes.

Portable working identity should become a first-class artifact that sits beside
canonical memory, not instead of it.

## Behavioral Wiring

### Integration Surface Model

Define three capability classes.

| Surface | Purpose | Typical Clients | Risk Level |
|---------|---------|-----------------|-----------|
| `brain-memory` | Capture + query the brain only | Claude Code, Codex, Cursor, first-party HAETSAL agents | Lowest |
| `brain-sources-read` | Read from connected source systems and selectively capture into the brain | Google ingestion workers, future source connectors, possibly trusted first-party agents | Medium |
| `brain-actions` | Mutate external systems | Future CoS / ops agents only | Highest |

Rules:

- `brain-memory` is the first implementation target.
- `brain-sources-read` is the second.
- `brain-actions` is explicitly delayed until the lower-risk surfaces are
  proven.

### Client Classes

| Client Class | Connection Pattern | Initial Scope |
|--------------|--------------------|---------------|
| MCP-native coding clients | Direct remote MCP | `brain-memory` |
| First-party HAETSAL agents | Internal MCP / trusted Worker path | `brain-memory`, later selective `brain-sources-read` |
| Web AI clients without first-class MCP | BYOC bundle + import/export recipe | portable profile + optional explicit exports |
| Source connectors | Worker / queue / webhook ingestion path | `brain-sources-read` |

### Source Classes

| Source Class | Pattern | Examples |
|--------------|---------|----------|
| Event-driven source read | Webhook/push -> queue fetch -> canonical capture | Gmail, Calendar |
| Explicit file/folder ingestion | Poll or event + explicit inclusion rules | Drive, Docs, Obsidian bridge |
| Historical import | Bounded batch import with lower-trust provenance | Gmail history, Calendar history, Drive history |
| External action | Deferred | Send email, create events, edit docs |

### Auth / Policy Direction

Current HAETSAL Access-gated auth remains acceptable for private/internal use.

Future direction should be:

- OAuth-ready / remote-MCP-friendly posture for external clients
- capability scopes attached to integration surfaces
- separation between:
  - tenant / identity auth
  - surface capability auth
  - source-provider OAuth grants

This session defines that target posture; it does not require a full auth
rewrite.

## Part 2: Integration Contract

### 2.1 `brain-memory`

Purpose:

- capture memory
- search/query memory
- semantic recall
- graph/timeline reads
- context preparation for trusted agents

Allowed:

- `capture_memory`
- `search_memory`
- `get_recent_memories`
- `get_document`
- `memory_status`
- `memory_stats`
- future `prepare_context_for_agent`

Forbidden:

- direct Gmail / Calendar / Drive mutation
- hidden source-system traversal outside the canonical brain contract

This is the default surface for:

- Claude Code
- Codex
- Cursor
- future MCP-native coding or operator tools

### 2.2 `brain-sources-read`

Purpose:

- read connected source systems
- transform source events/documents into canonical capture
- allow trusted first-party agents to assemble richer context from source data

Allowed:

- Gmail read
- Calendar read
- Drive/Docs read
- source metadata fetch
- selective canonical capture from those sources

Forbidden:

- send email
- create/update calendar events
- edit docs/files

### 2.3 `brain-actions`

Purpose:

- external system mutation

Allowed later:

- send email
- create/update events
- edit docs or task systems

Default stance:

- not part of the first integration rollout
- explicit human-in-the-loop bias
- stronger review/approval boundary

## Part 3: BYOC / Portable Context Bundle

HAETSAL should adopt a BYOC-style portable working-identity artifact alongside
its canonical memory substrate.

This bundle should not replace canonical memory. It should provide a portable,
reviewable summary of:

- how the user works
- communication preferences
- workflow preferences
- recurring rhythms
- durable project/person context
- what an assistant should and should not do

Recommended initial artifact family:

- `operating-model.json`
- `USER.md`
- `SOUL.md`
- `HEARTBEAT.md`
- `schedule-recommendations.json`

Rules:

- artifacts are exportable and user-owned
- they may be re-imported or pulled by MCP-capable clients
- they should be derivable from canonical memory plus explicit review
- they are not treated as the only truth about the user

Use cases:

- migrating between AI clients
- supplying context to non-MCP web clients
- separating personal working identity from employer/project data
- bootstrapping first-party agents such as Chief of Staff

## Part 4: Source Ingestion Model

### 4.1 Gmail

Model:

- event-driven receipt
- queue fetch of the actual thread
- selective capture into canonical memory
- source remains authoritative for the mailbox

Rules:

- do not mirror the full mailbox into HAETSAL
- preserve provenance back to thread/message ids
- capture only the memory representation needed for the brain

### 4.2 Calendar

Model:

- event-driven updates
- selective capture of useful event context
- source remains authoritative for the calendar objects

Rules:

- avoid overcapturing transient noise
- preserve provenance to source event ids

### 4.3 Drive / Docs / Obsidian Bridge

Model:

- explicit inclusion rules or narrow event-driven capture
- support user-controlled folders/tags such as `/to-brain/` or metadata markers

Rules:

- do not blindly ingest every document
- keep `/from-brain/`-style output shelves excluded from re-ingestion

### 4.4 Historical Import

Model:

- bounded bootstrap import
- lower-trust provenance class
- separate operational lane from live event ingestion

Rules:

- historical import is for initial seeding, not continuous replay
- imported material should remain distinguishable from live/user-authored
  capture

## Part 5: Provenance / Trust Classes

Define at least these canonical provenance classes:

| Class | Meaning |
|-------|---------|
| `user_authored` | Direct human-authored memory or note |
| `agent_authored` | First-party agent output intentionally captured |
| `source_ingested` | Selective live source-system capture |
| `bootstrap_import` | Historical bounded import |
| `byoc_export` | Portable working-identity artifact derived from reviewed context |

The system should be able to answer:

- was this directly authored?
- was it pulled from a source system?
- was it historical import?
- is it part of the portable identity bundle?

## Part 6: Cloudflare Delivery Posture

This session should align HAETSAL toward Cloudflare's current MCP direction
without forcing an unnecessary rewrite.

### Immediate stance

- keep HAETSAL's existing Worker / McpAgent shell
- keep the current canonical memory substrate and tool contracts
- do not split into separate backends per AI vendor

### Directional stance

- modern remote MCP transport posture
- OAuth-ready external-client model
- narrower scoped surfaces rather than one universal tool blast radius
- evaluate portal / Code mode only when the tool surface grows large enough to
  justify it

Interpretation:

- Cloudflare's guidance should influence **delivery shape and permission scope**
- Nate/OB1's guidance should influence **one-brain portability and integration
  philosophy**

These are complementary, not contradictory.
