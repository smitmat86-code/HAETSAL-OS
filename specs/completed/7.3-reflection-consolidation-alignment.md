## What We're Building

Align current Hindsight reflection/consolidation behavior with the canonical
open-brain architecture.

Today HAETSAL’s consolidation/reflection behavior still carries pre-canonical
assumptions from the earlier Hindsight-first architecture. Session 7.3 makes
reflection/consolidation an explicit part of the canonical operation model by:

- attaching reflection/consolidation work to canonical records and operations
- tying semantic projection observability back to the canonical status surface
- preserving the current Hindsight runtime while removing the old “out-of-band”
  feeling of consolidation state

When complete, semantic projection will no longer stop at capture + recall.
Its reflective/consolidation lifecycle will be visible and queryable as part of
the canonical open-brain system.

## THE Brain Laws Check

### Law 1: One Public Face

All reflection/consolidation alignment remains behind the current Worker /
McpAgent surface. This session may rewire internal cron/service/runtime paths,
but it must not add a public Hindsight maintenance endpoint or expose a direct
reflect API.

### Law 2: Zero-Knowledge

Reflection/consolidation alignment may only add metadata and status truth to D1.
No raw memory content may be copied into D1, KV, Analytics Engine, or queue
payloads for reflection bookkeeping. Any engine calls that still require access
to memory content must continue to do so through the existing trusted Hindsight
runtime boundary.

### Law 3: Agent Write Policy

This session affects background semantic lifecycle behavior, not user/agent
write permissions. It must not widen write policy or create a new way for
agents to write procedural content through reflection bookkeeping.

|-------|-------|-------------|-----------|

| Reflection/consolidation failed | canonical audit lane | `memory.projection.hindsight_reflect_failed` | Atomic with canonical reconciliation write |

Audit must stay truthful. Do not imply semantic projection completion if only
reflection has begun, and do not imply reflection completion if the engine work

| Graph projection / temporal alignment | Deferred to Phase 8 | Out of scope |

### Other Behavioral Wiring

- Reuse the current HAETSAL shell and existing Hindsight reflect/consolidation

  attach current reflection behavior to canonical operations/status.
- Canonical memory status should become the truthful place to understand:
  - capture accepted/queued/completed
  - semantic projection state
  - reflection/consolidation state

### Migration Decision

- Session 7.3 reused the existing canonical operation + projection model and
  attached reflection/consolidation observability through canonical audit rows
  plus the existing `consolidation_runs` table.

### Exact Canonical Status Additions Shipped

- `memory_status` now returns a top-level `reflection` subsection:
  - `mode: 'hindsight'`
  - `status: 'pending' | 'queued' | 'completed' | 'failed'`
  - `targetRef?: string | null`
  - `updatedAt?: number | null`
  - `errorMessage?: string | null`
- `pending` means the Hindsight semantic projection is already usable, but no
  reflection/consolidation cycle has been recorded yet for that operation.
- `queued`, `completed`, and `failed` are derived truthfully from canonical
  reflection audit events, with `targetRef` pointing at the corresponding
  Hindsight bank consolidation run when the run record exists.

### How Existing Hindsight Reflection / Consolidation Was Attached

- The existing 3.3 consolidation scheduler in `src/cron/consolidation.ts`
  remains the driver.
- A narrow alignment layer in
  `src/services/canonical-hindsight-reflection.ts` now:
  - finds completed Hindsight-backed canonical operations that still need a
    reflection/consolidation lifecycle recorded
  - writes canonical audit events
    `memory.projection.hindsight_reflect_started`,
    `memory.projection.hindsight_reflect_completed`, and
    `memory.projection.hindsight_reflect_failed`
  - lets `memory_status` project that lifecycle back onto the canonical
    operation without changing public HTTP routes or storing raw memory bodies
- This keeps reflection/consolidation tied to canonical operations instead of
  leaving it visible only through Hindsight runtime behavior.
