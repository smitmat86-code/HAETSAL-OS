## What We're Building

Fan out canonical capture events into Graphiti using the design and mapping
rules established in Session 8.1.

This session makes Graphiti a real canonical projection target by:

- implementing the Graphiti projection worker/runtime seam
- projecting canonical captures into Graphiti episode/entity/edge operations
- recording canonical-to-graph identity mappings
- surfacing graph projection lifecycle truthfully through the canonical
  operation/status model

When complete, Graphiti will stand beside Hindsight as a live projection target
fed from canonical capture truth, while still remaining behind the canonical
public interface.

---

## THE Brain Laws Check

### Law 1: One Public Face

All Graphiti ingestion remains behind the current Worker / McpAgent surface.
This session may add internal queue/worker/runtime integration, but it must not
introduce a new public Graphiti route or let clients write directly to Graphiti.

### Law 2: Zero-Knowledge

Canonical capture remains the source of truth. Queue payloads, D1 projection
rows, mapping tables, and audit records must not contain raw memory content.
If Graphiti needs content or extracted structure, that must be resolved inside
the trusted projection runtime boundary, not by copying content into D1/KV/
Analytics for convenience.

### Law 3: Agent Write Policy

This session changes graph projection plumbing, not write policy. It must not
create any new direct write path to Graphiti or widen procedural-write
permissions.

| Event | Table | Action String | Blocking? |
|-------|-------|-------------|-----------|
| Graphiti projection submission started | canonical audit lane | `memory.projection.graphiti_started` | Atomic with projection result/state write |
| Graphiti projection queued/accepted | canonical audit lane | `memory.projection.graphiti_queued` | Atomic with canonical reconciliation write |

| Graphiti projection failed | canonical audit lane | `memory.projection.graphiti_failed` | Atomic with canonical result/state write |

handoff occurred, and do not hide failure/retry state inside engine-local logs.

### Compute Routing

| Operation | Tier | Rationale |
|-----------|------|-----------|
| Canonical capture acceptance | Sync Worker / service | Already delivered in 6.3 |
| Projection dispatch enqueue | Queue C3 | Durable fan-out boundary |
| Graphiti projection execution | Queue consumer / internal runtime | Should not block MCP response |
| Graphiti reconciliation/status updates | Sync Worker / D1 service | Lightweight metadata truth |
| Graph query surface | Deferred to Session 8.3 | Out of scope |

### Other Behavioral Wiring

- Graphiti must be fed from canonical capture events, not directly from clients.
- Reuse the 8.1 deployment posture and mapping contract.
- Graphiti projection should follow the same high-level lifecycle pattern as
  Hindsight where useful:
  - accepted
  - queued
  - completed
  - failed
  But do not force false symmetry if graph-specific reconciliation needs extra
  identity handling.
- Canonical memory status should surface graph projection truth alongside the
  existing semantic projection truth.

### Migration Decision

- `1018_graphiti_ingestion_projection.sql` **was required**.
- The existing canonical projection job/result tables were sufficient for graph
  lifecycle status, but not sufficient for truthful canonical-to-Graphiti
  identity mapping across episode/entity/edge refs.
- Session 8.2 adds `canonical_graph_identity_mappings` with a projection-job
  unique key and canonical-key lookup index. No raw memory content is stored in
  this table.

### Exact Graphiti Runtime / Posture Used

- Session 8.2 keeps the 8.1 staged posture:
  **trusted external Graphiti service first, Cloudflare queue shell later
  movable to Containers**.
- The live runtime seam is an internal POST from the queue consumer to
  `${GRAPHITI_API_URL}/v1/canonical/projections` with optional bearer auth from
  `GRAPHITI_API_TOKEN`.
- No public Graphiti HTTP route was added; clients still write only to the
  canonical capture path.

### Canonical-to-Graph Identity Mapping Shipped

- Each `graphiti` projection job now builds the 8.1 plan from canonical ids plus
  a KEK-encrypted Graphiti payload stored in R2.
- One Graphiti episode is anchored to
  `canonical://captures/{captureId}` and persisted as `graph_kind = 'episode'`.
- Entities persist with their 8.1 deterministic `canonicalKey` values and
  `graph_kind = 'entity'`.
- Edges now have deterministic canonical keys and persist as
  `graph_kind = 'edge'`.
- Mapping rows are stored in `canonical_graph_identity_mappings` and keyed by
  `projection_job_id + canonical_key + graph_kind`, so retries update the same
  identity lane instead of inventing duplicate canonical anchors.
- `canonical_projection_results.target_ref` stores the primary Graphiti target
  ref, while `engine_operation_id` carries any queued Graphiti operation ref.

### Graph Lifecycle Behavior Shipped

- Canonical queue dispatch remains metadata-only and unchanged at the public
  boundary.
- The ingestion consumer now fans out both `hindsight` and `graphiti`
  projection jobs from the same dispatch message.
- When Graphiti runtime is configured, the projection lane records:
  `graphiti_started` -> `graphiti_queued` or `graphiti_completed` ->
  `graphiti_failed` as needed.
- When Graphiti runtime is **not** configured, the consumer leaves the graph
  job in its queued canonical state instead of manufacturing a fake engine
  failure. This keeps lifecycle truth coherent in environments that have not
  enabled the external runtime yet.

### Deviations From Planned Scope

- The planned separate `canonical-graphiti-consumer.ts` file was not needed.
  Session 8.2 reused the existing `canonical-projection-consumer.ts` dispatch
  seam and added Graphiti routing there.
- `src/services/canonical-graphiti-payload.ts` was added beyond the planned
  file list so Graphiti could follow the existing KEK-encrypted payload pattern
  already used by the Hindsight adapter, without copying raw memory content
  into D1/KV/queue payloads.
- The graph lane remains configuration-gated by `GRAPHITI_API_URL` instead of
  forcing every environment to fail graph jobs immediately. This was the
  narrowest choice that kept canonical lifecycle semantics truthful while
  preserving existing unconfigured test and dev environments.
