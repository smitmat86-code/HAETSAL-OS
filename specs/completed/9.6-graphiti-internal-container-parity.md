## What We're Building

Move Graphiti/Kuzu inside HAETSAL as an internal container-backed runtime and
retire the current production dependency on an external Graphiti HTTP service.

This session changes the Graphiti runtime posture from:

- trusted external Graphiti service first

to:

- internal HAETSAL-managed Graphiti container runtime

The goal is not to invent a new graph architecture. The goal is to finish the
next intended stage of the existing architecture:

- canonical Worker / queue shell remains the public orchestration layer
- Graphiti remains an internal projection/query engine
- Kuzu runs in a containerized runtime owned by HAETSAL

---

## Why This Session Exists

Current production behavior proves two things:

1. Hindsight is now healthy again and semantically retrievable through the
   canonical interface.
2. Graphiti is not actually live in production because the Worker is still
   configuration-gated on `GRAPHITI_API_URL` / `GRAPHITI_API_TOKEN`, and those
   runtime values are not present.

That means Graphiti currently has:

- canonical schema
- canonical job/result lifecycle
- query tools
- graph identity mappings

but not a real internal engine runtime in HAETSAL production.

This session closes that gap.

---

## Strategic Answer

### Does Graphiti still need an API?

It still needs a runtime interface, but not a public external API.

The Worker must still talk to a graph engine somehow. In the internalized
model, that interface becomes:

- HAETSAL internal container boundary

not:

- external public Graphiti service

### Should this match Hindsight?

Yes, substantially.

That similarity is not just aesthetic preference. It creates real value:

- same deployment and ops mental model
- same Worker-to-container trust boundary
- same health/readiness and cold-start patterns
- same failure-handling expectations
- same observability and debugging playbook

We should seek **operational parity**, not forced byte-for-byte symmetry.

### Should this be Python or Rust?

Default recommendation:

- internalize the current Graphiti/Kuzu runtime first in the smallest viable
  language/runtime path
- do not make "move to Rust" part of the first migration unless the existing
  runtime cannot be containerized safely

Rust may become a later optimization or replacement path, but it is not the
smallest route to container parity.

---

## THE Brain Laws Check

### Law 1: One Public Face

Graphiti must remain internal to HAETSAL. No new public Graphiti route or
client-facing Graphiti API surface should appear.

### Law 2: Zero-Knowledge

The Worker / D1 / queues / analytics stores must still avoid persisting raw
memory content outside the trusted canonical capture and trusted runtime path.
Containerization must not become an excuse to duplicate raw content into new
operational stores.

### Law 3: Agent Write Policy

This session changes internal graph runtime topology. It must not add any new
public write surface or broaden client authority.

---

## Target Runtime Shape

### Near-Term Target

Graphiti becomes an internal HAETSAL container binding, analogous to Hindsight:

- Worker receives canonical projection dispatch
- Worker calls internal Graphiti runtime
- Graphiti writes/returns graph refs and identity mappings
- canonical query tools continue to read through the canonical layer

### Recommended Shape

Start with:

- **single Graphiti container**

Then allow a later fast follow to:

- **API container + dedicated Graphiti worker container**

if ingestion, indexing, or query concurrency requires separation.

### Similarity To Hindsight

We should match Hindsight in these ways:

- internal container binding
- DO/container lifecycle ownership in HAETSAL
- explicit runtime health/readiness seam
- env-driven model/runtime config
- no public engine route

We do **not** have to force exact symmetry where the engines differ internally.

## Migration Plan

### Phase A: Transport Seam

Refactor Graphiti submission behind a small internal client abstraction.

Expected addition:

- `src/services/graphiti-client.ts`

Responsibilities:

- define the internal Graphiti runtime contract
- hide whether the runtime is external or container-backed
- provide:
  - `submitCanonicalProjection(...)`
  - `getProjectionStatus(...)`
  - optional `healthcheck()`

This phase should preserve current behavior while preparing clean cutover.

### Phase B: Internal Graphiti Container

Add a new container class under the Worker runtime, analogous to Hindsight.

Expected additions:

- `src/workers/mcpagent/do/GraphitiContainer.ts`
- `wrangler.toml` container + DO bindings
- export from `src/workers/mcpagent/index.ts`

Responsibilities:

- run Graphiti/Kuzu inside HAETSAL
- expose only an internal runtime interface
- provide a health endpoint/readiness contract the Worker can trust

No public route should be added.

### Phase C: Container-Backed Client

Implement the container backend in `graphiti-client.ts`.

The Worker should be able to call the Graphiti container through a trusted
internal boundary rather than `GRAPHITI_API_URL`.

This should become the intended production path.

### Phase D: Container-First Runtime Mode

Introduce an explicit runtime mode, for example:

- `GRAPHITI_RUNTIME_MODE = "container" | "external"`

Default intended production value:

- `container`

External mode may remain temporarily for transition/testing only.

### Phase E: Truthful Failure Semantics

Once container mode is the intended prod path:

- Graphiti jobs must no longer remain silently `queued` forever just because no
  runtime is configured
- if container mode is required and the runtime is unreachable, graph jobs
  should fail truthfully with actionable error state

This is important operationally. Silent queue drift is no longer acceptable.

### Phase F: Optional Fast Follow - Dedicated Graphiti Worker Topology

After initial container parity is working, evaluate whether Graphiti should
match Hindsight even more closely with:

- a dedicated Graphiti worker container count
- split API/query vs worker/ingestion topology

This is a fast follow only if needed for runtime health or scale.

---

## Why Hindsight Parity Matters

Matching Hindsight has real value beyond preference.

### Operational Value

- one mental model for engine runtimes
- one playbook for cold starts, readiness, and health
- one debugging pattern for Worker -> DO/container -> engine
- less drift between "semantic engine" and "graph engine" ops

### Engineering Value

- reusable deployment/config patterns
- reusable tests and smoke patterns
- easier onboarding for future work
- less chance of one engine silently drifting into "special-case" territory

### Boundaries

Parity should mean:

- same trust boundary
- same platform ownership
- same observability posture

not:

- identical internals regardless of engine needs

So yes, parity is desirable here, and not merely stylistic.

## Part 3: Internal Runtime Contract

The internal Graphiti client should define a narrow contract similar to:

```typescript
export interface InternalGraphitiProjectionRequest {
  tenantId: string
  projectionJobId: string
  captureId: string
  operationId: string
  documentId: string
  posture: string
  plan: Record<string, unknown>
  content: { body: string }
}

export interface InternalGraphitiProjectionResponse {
  status: 'queued' | 'completed'
  targetRef?: string | null
  operationRef?: string | null
  episodeRefs?: string[]
  entityRefs?: string[]
  edgeRefs?: string[]
  mappings?: Array<{
    canonicalKey: string
    graphRef: string
    graphKind: 'episode' | 'entity' | 'edge'
  }>
}
```

The canonical projection layer should not care whether this response came from
an internal container or an older external transport.

## Part 4: Submission / Reconciliation Behavior

### Submission

`submitGraphitiProjection(...)` should:

- load canonical payload as it does today
- call the internal Graphiti client
- persist result refs/mappings as it does today

### Reconciliation

If Graphiti uses asynchronous completion:

- add an internal status/reconciliation path analogous to Hindsight

If Graphiti completes synchronously enough:

- preserve the smallest truthful completion path

Do not invent fake completion. Do not leave jobs permanently queued when the
runtime is required and missing.

    fallback when `GRAPHITI_RUNTIME_MODE=external`.

  - Ship a **single internal Graphiti container** first.
  - Do not split ingestion/query topology in 9.6.
  - Match Hindsight where operationally useful: internal ownership, Worker
    runtime boundary, readiness probe, and truthful lifecycle semantics.

- **Deviations From Planned Scope**

    Kuzu, not a direct exposure of Graphiti's broader upstream public API
    surface.
  - External mode was not fully removed because a narrow explicit migration
    fallback is still useful during rollout and test transitions.
  - Durable Kuzu persistence across container recreation is **not** completed in
    this session; the current implementation uses container-local storage and
    should be treated as the smallest internal runtime step, not the final
    durability story.
  - The Worker and Graphiti container were deployed, but end-to-end live proof
    for fresh captures remains **not fully confirmed** because the production
    invocation surface is behind Cloudflare Access / signed webhook flows and no
    non-interactive capture path was available from this workspace to submit and
