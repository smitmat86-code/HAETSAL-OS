# Implementation Plan: Phase 2.3 — Browser Rendering + First Write Surfaces

## Goal

Make actions real: `brain_v1_act_browse` navigates a URL via Cloudflare Browser Rendering (Puppeteer), `brain_v1_act_create_event` and `brain_v1_act_modify_event` call the Google Calendar API. Add automatic episodic memory on successful execution and a 5-minute undo window for reversible actions.

## User Review Required

> [!IMPORTANT]
> **`@cloudflare/puppeteer` installation.** This is a new npm dependency. Must use `@cloudflare/puppeteer` (not `puppeteer` directly). Need to pin to the latest compatible version before implementation. Verify package exists on npm and check for known issues with the BROWSER binding in local dev.

> [!WARNING]
> **BROWSER binding in tests.** The BROWSER binding cannot run a real browser in `vitest-pool-workers`. Tests for `executeBrowse()` must stub the binding. Real browse behavior validation requires manual testing with `wrangler dev`. This is documented in the spec.

> [!IMPORTANT]
> **Undo window background expiry.** After 5 minutes, `completed_reversible` → `completed`. The spec mentions "Phase 1.4 cron or explicit state machine" as a TODO. For 2.3, we should implement a lightweight approach — either a D1 query in the existing cron trigger or a `setAlarm()` on the DO (since DO alarms are available with SQLite-backed DOs).

### Undo Route

#### [MODIFY] [index.ts](file:///c:/Users/matth/Documents/HAETSAL%20OS/src/workers/mcpagent/index.ts)
- Add `POST /actions/:id/undo` route (behind CF Access auth)
- Validation: action must be in `completed_reversible` state AND within 5-minute window
- Within window: call integration-specific undo (e.g., delete created calendar event)
- State transition: `completed_reversible` → `undone`
- After window: return 409 Conflict
- Atomic D1 batch: state UPDATE + action_audit INSERT (`action.undone`)
- WebSocket push on undo
