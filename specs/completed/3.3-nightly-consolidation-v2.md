## What Changed From v1 (Read This First)

The original spec had 6 passes. After auditing the full Hindsight v0.4.16 API,
two passes are fully replaced by native Hindsight capabilities:

| Pass | v1 | v2 |
|------|----|----|
| Pass 1 — Fact Distillation | Custom LLM → semantic retain | **REMOVED** — Hindsight does this natively via `/consolidate` + `observations_mission` |
| Pass 2 — Contradiction Resolution | Blind recall + LLM | **KEPT** — simplified using `/memories/{id}/history` to detect observation reversals |
| Pass 3 — Bridge Edges | Blind recall → LLM | **KEPT** — seeded from `/graph` endpoint instead of recall, cheaper and more accurate |
| Pass 4 — Behavioral Patterns | Custom LLM → procedural | **UNCHANGED** — no Hindsight equivalent |
| Pass 5 — Mental Models | LLM → semantic with `is_mental_model: true` | **REMOVED** — Hindsight Mental Models API with `refresh_after_consolidation: true` replaces entirely |
| Pass 6 — Gap Questions | Raw LLM call | **KEPT** — now uses `reflect` with `response_schema` (structured JSON output) instead of raw LLM |

**Result: 4 passes instead of 6. Cron is leaner, more correct, and avoids
duplicating work Hindsight already does better.**

Three additional architectural changes:
1. **Webhook-driven trigger** — Hindsight fires `consolidation.completed` events.
   Our cron now reacts to this webhook rather than running on a fixed timer. The
   `0 2 * * *` cron becomes a fallback for nights where Hindsight doesn't fire.
2. **`BaseAgent.open()` change** — No longer recalls with `is_mental_model: true`
   metadata filter. Now calls `GET /mental-models?tags=domain:{domain}` directly.
3. **Bootstrap additions** — `2.4` must create one Mental Model per domain and set
   `observations_mission` on the bank. Document this as a 2.4 as-built addendum.

## What We're Building

The brain's sleep cycle. While the tenant is away, the nightly cron does the
deep work real-time sessions can't: it detects contradictions, finds bridge edges
across domains, extracts behavioral patterns, and surfaces gap questions for the
morning brief.

**Hindsight now handles fact distillation and mental model synthesis automatically.**
We configure it once (in 2.4 bootstrap) and it runs continuously. Our cron focuses
on the genuinely custom work: cross-domain bridge discovery, behavioral pattern
extraction, contradiction flagging, and gap preparation.

## THE Brain Laws Check

### Law 1: One Public Face
New webhook receiver route: `POST /hindsight/webhook` — added before auth
middleware (same pattern as Telnyx SMS and Telegram). Validated via HMAC-SHA256
secret. This is a necessary exception, not a violation — same as prior precedents.

### Law 2: Zero-Knowledge (CRITICAL)

**Cron KEK flow (unchanged from v1):**
The KEK stored in KV at `cron_kek_raw:{tenantId}` is raw AES-256-GCM key bytes
(base64). NOT encrypted with TMK. The cron imports these bytes as a CryptoKey and
uses it to encrypt/decrypt Hindsight calls.

Security model: if KV is compromised, attacker can read memories for ≤24h after
tenant's last session. Accepted tradeoff — documented in `LESSONS.md`.

**New: Webhook validation**
Hindsight webhook payload is validated via HMAC-SHA256 using `HINDSIGHT_WEBHOOK_SECRET`.
Reject any request without valid signature — same as Telnyx/Telegram pattern.

### Law 3: Agent Write Policy
`consolidationRetain()` remains the SOLE procedural write path.
Exists only in `src/cron/passes/pass4-patterns.ts`.
Not exported outside `src/cron/`.

## Webhook Route (New — index.ts)

```typescript
// BEFORE auth middleware — same as SMS/Telegram pattern
app.post('/hindsight/webhook', async (c) => {
  // Validate HMAC-SHA256 signature
  const sig = c.req.header('X-Hindsight-Signature')
  const body = await c.req.text()
  const expected = await computeHmac(body, c.env.HINDSIGHT_WEBHOOK_SECRET)
  if (sig !== expected) return c.json({}, 403)

  const payload = JSON.parse(body)
  if (payload.event_type === 'consolidation.completed') {
    const tenantId = payload.bank_id  // bank_id maps to hindsight_tenant_id in D1
    // Trigger our 4-pass cron for this specific tenant
    c.executionCtx.waitUntil(runConsolidationPasses(tenantId, c.env, c.executionCtx))
  }

  return c.json({ ok: true })
})
```

## The 4 Passes

Each pass runs sequentially. `Promise.allSettled` for per-item within each pass.
All Hindsight calls use the tenant's KEK (derived from `cron_kek_raw:{tenantId}`).

---

### Pass 2 — Contradiction Detection (renamed from Pass 2)

**Goal:** Find semantic memories where Hindsight's own observation history shows
a reversal — a sign that newer evidence contradicted an older observation.

```typescript
// Use Hindsight's /memories/list to get semantic memories, limit 50
// For each memory: fetch /memories/{id}/history
// If history shows 2+ entries where content significantly differs → potential contradiction
//
// Batch contradictions into LLM call:
// "Review these memory pairs where the observation changed significantly.
//  For each: determine if it's a genuine contradiction or a natural update.
//  Return JSON: { contradictions: Array<{
//    memory_id: string,
//    resolution: 'genuine_contradiction' | 'natural_update' | 'ambiguous',
//    reasoning: string
//  }> }"
//
// For 'genuine_contradiction': write anomaly_signal type 'memory.contradiction'
//   with memory_id in metadata — Hindsight supersession is Phase 4+
// For 'ambiguous': write anomaly_signal type 'memory.contradiction_unresolved'
// For 'natural_update': no action — expected behavior

// Uses /memories/list + per-memory /history — NOT a recall call
// No encryption/decryption needed (we're querying structure, not content)
// NOTE: If memory content is needed for LLM, fetch and decrypt with KEK
```

**Why this is better than v1:** Hindsight's observation history already tracks
what changed and when. We use structural signal (change occurred) to filter
candidates before the expensive LLM call — rather than scanning all 50 memories.

---

### Pass 3 — Bridge Edges (seeded from /graph)

**Goal:** Find non-obvious connections across domains. Surface as morning brief gifts.

```typescript
// Fetch graph topology from Hindsight (no decryption needed — structural only)
const graphRes = await env.HINDSIGHT.fetch(
  `/v1/default/banks/${bankId}/graph?limit=200`
)
const { nodes, edges } = await graphRes.json()

// Filter: nodes from different domains (infer domain from tags)
// Find nodes with NO existing edge between them but semantic neighbors in common
// (Structural hole detection — nodes that share a neighbor but aren't connected)
// Take top 10 candidate pairs by shared-neighbor count

// For each candidate pair: fetch and decrypt actual content with KEK
// Batch into single LLM call:
// "These memory pairs share indirect connections but no direct edge.
//  Identify which pairs reveal a genuine cross-domain insight.
//  Return JSON: { bridges: Array<{
//    memory_id_a: string, memory_id_b: string,
//    insight: string,   // The connection in one clear sentence
//    domains: [string, string]
//  }> }
//  Max 5 bridges. Quality over quantity."
//
// For each bridge:
//   consolidationRetain(insight, 'semantic', 'general', 'pass3_bridge')
//   metadata: { is_bridge: true, bridge_memory_ids: [id_a, id_b] }
//
// Bridges appear as "gifts" in morning brief (3.4 reads is_bridge: true)

// KEY ADVANTAGE over v1: /graph is fast, structural, and requires no decryption.
// We only decrypt the small set of candidate pairs — not all 40 memories.
```

### Pass 4 — Behavioral Pattern Extraction (SOLE PROCEDURAL WRITE PATH)

**Unchanged from v1.**

```typescript
// Recall: episodic memories, session provenance, last 30 days, limit 50
// Decrypt with KEK

// LLM: "Extract behavioral patterns from these session memories.
//   A behavioral pattern is a recurring tendency visible across multiple sessions.
//   Return JSON: { patterns: Array<{
//     pattern: string,       // 1-2 sentence description
//     confidence: number,    // 0.0-1.0
//     domain: string,        // which life domain this pattern lives in
//     evidence_count: number // how many sessions show this
//   }> }
//   Only return patterns with confidence > 0.6.
//   Max 3 patterns per run."
//
// For each pattern with confidence > 0.6:
//   consolidationRetain(pattern.pattern, 'procedural', pattern.domain, 'pass4_behavioral')
//
// THIS IS THE ONLY PLACE IN THE ENTIRE CODEBASE THAT WRITES memory_type: 'procedural'
// consolidationRetain() accepts procedural — retainContent() does not

// Audit: pattern count, domains covered
```

**Note:** `consolidationRetain()` is a module-private function in
`src/cron/passes/pass4-patterns.ts`. It is NOT exported. It calls `retainContent()`
internally with an extended type union that includes 'procedural'. The only way to
write procedural memories is through this function, in this file.

### Pass 4 (renamed) — Gap Identification

**Uses `reflect` with structured output (replaces raw LLM call from v1 Pass 6).**

```typescript
// Call Hindsight reflect endpoint with response_schema for structured output
const reflectRes = await env.HINDSIGHT.fetch(
  `/v1/default/banks/${bankId}/reflect`,
  {
    method: 'POST',
    body: JSON.stringify({
      query: `What important questions about this person remain unanswered?
        What decisions are pending or deferred? What areas seem underexplored
        given what we know?`,
      budget: 'mid',
      response_schema: {
        type: 'object',
        required: ['gaps'],
        properties: {
          gaps: {
            type: 'array',
            items: {
              type: 'object',
              required: ['question', 'domain', 'priority'],
              properties: {
                question: { type: 'string' },
                domain: { type: 'string' },
                priority: { type: 'string', enum: ['high', 'medium', 'low'] }
              }
            }
          }
        }
      },
      tags: [`tenant:${tenantId}`],
    })
  }
)
const { structured_output } = await reflectRes.json()

// Write top 3 gaps to consolidation_gaps D1 table — NOT Hindsight
// D1 only: gaps are metadata (what to ask), not content (what we know)
// No encryption — no PII in gap questions, structural only
for (const gap of structured_output.gaps.slice(0, 3)) {
  await env.D1_US.prepare(`
    INSERT INTO consolidation_gaps (id, tenant_id, run_id, question, domain, priority, surfaced, created_at)
    VALUES (?, ?, ?, ?, ?, ?, 0, ?)
  `).bind(crypto.randomUUID(), tenantId, runId, gap.question, gap.domain, gap.priority, Date.now()).run()
}

// Gap questions surfaced in morning brief (3.4 reads surfaced=0, high priority first)
```

**Why this is better than v1:** `reflect` uses Hindsight's full retrieval stack
(semantic + BM25 + entity graph + temporal) to find gaps — not just a raw LLM
call on whatever we happened to recall. Better signal, less prompt engineering.

## D1 Migration

### [NEW] `migrations/1008_brain_consolidation.sql`

```sql
CREATE TABLE IF NOT EXISTS consolidation_runs (
  id                    TEXT PRIMARY KEY,
  tenant_id             TEXT NOT NULL,
  started_at            INTEGER NOT NULL,
  completed_at          INTEGER,
  status                TEXT NOT NULL DEFAULT 'running',
  trigger               TEXT NOT NULL DEFAULT 'cron',  -- 'cron' | 'webhook'
  pass1_contradictions  INTEGER NOT NULL DEFAULT 0,
  pass2_bridges         INTEGER NOT NULL DEFAULT 0,
  pass3_patterns        INTEGER NOT NULL DEFAULT 0,
  pass4_gaps            INTEGER NOT NULL DEFAULT 0,
  error_message         TEXT,
  FOREIGN KEY (tenant_id) REFERENCES tenants(id)
);

CREATE UNIQUE INDEX IF NOT EXISTS idx_consolidation_runs_dedup
  ON consolidation_runs(tenant_id, (started_at / 86400000));

CREATE INDEX IF NOT EXISTS idx_consolidation_runs_tenant
  ON consolidation_runs(tenant_id, started_at DESC);

CREATE TABLE IF NOT EXISTS consolidation_gaps (
  id          TEXT PRIMARY KEY,
  tenant_id   TEXT NOT NULL,
  run_id      TEXT NOT NULL,
  question    TEXT NOT NULL,
  domain      TEXT NOT NULL,
  priority    TEXT NOT NULL,
  surfaced    INTEGER NOT NULL DEFAULT 0,
  created_at  INTEGER NOT NULL,
  FOREIGN KEY (tenant_id) REFERENCES tenants(id),
  FOREIGN KEY (run_id) REFERENCES consolidation_runs(id)
);

CREATE INDEX IF NOT EXISTS idx_consolidation_gaps_tenant
  ON consolidation_gaps(tenant_id, surfaced, priority);
```

The `UNIQUE INDEX` on `(tenant_id, started_at / 86400000)` is the deduplication
guard — prevents both webhook and cron from running consolidation twice in one day.
