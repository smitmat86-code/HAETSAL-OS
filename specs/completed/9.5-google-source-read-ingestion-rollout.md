## What We're Building

Roll out the first real external **source-read** surface for the open brain:

- `brain-sources-read`

This session brings Google into the open-brain architecture as a read-only,
selective, provenance-rich source family:

- Gmail
- Calendar
- Drive / Docs

The goal is to let the brain learn from Google's signals and documents without:

- pretending Google is just another memory client
- mirroring all Google data into HAETSAL
- granting external-source write authority too early

This is the first high-value source-system rollout after `brain-memory` has
already established the memory-only client surface.

---

## Why This Session Exists

`brain-memory` solves:

- how MCP-native clients read/write to the brain itself

But it does not solve:

- how the brain learns from source systems like Gmail, Calendar, and Drive

Google is different from Codex / Claude Code / Cursor because it is not a brain
client. It is an external source of evidence and activity.

So this session defines and rolls out:

- how Google source-read works
- how selective capture works
- how provenance remains truthful
- how we avoid turning HAETSAL into a fragile mirrored shadow of Google

This session is read-only with respect to Google. No sends, edits, or event
creation belong here.

---

## THE Brain Laws Check

### Law 1: One Public Face

Google source ingestion must still feed the same canonical brain and canonical
capture pipeline. It may use source-specific Worker/webhook/queue internals,
but must not create a second memory substrate or a Google-only memory product.

### Law 2: Zero-Knowledge

Google source-read rollout must not justify dumping raw email/doc content into
D1, KV, Analytics Engine, or rollout-side caches. Canonical capture plus
R2/Postgres remain the durable truth.

### Law 3: Agent Write Policy

This session is Google read-only. It must not grant send-email,
create-calendar-event, or edit-doc authority under the banner of source
ingestion.

---

## Strategic Answer

### What does Google do in this architecture?

Google acts as a **source system**, not as a general-purpose brain client.

### What surface owns it?

- `brain-sources-read`

### Does this duplicate Google?

Not as a naive mirror.

The system should:

- selectively capture useful representations into canonical memory
- preserve provenance and source references
- leave Google authoritative for its native objects

### Does Google write come here?

No.

Google write/actions belong in a later `brain-actions` lane.

## Behavioral Wiring

### Integration Surface

This session rolls out only:

- `brain-sources-read`

It depends on the existence of `brain-memory`, but it does not blur the two
surfaces together.

### Source Classes

| Source | Pattern | Initial Mode |
|--------|---------|--------------|
| Gmail | event-driven selective thread capture | read-only |
| Calendar | event-driven selective event capture | read-only |
| Drive / Docs | explicit inclusion / narrow event or poll capture | read-only |

### Allowed Operations

Allowed in this rollout:

- receive Google source events or poll source state
- fetch source content through Google APIs
- selectively convert useful source material into canonical capture
- expose provenance/source references to the brain

Forbidden:

- send email
- archive email as an action surface
- create/update/delete calendar events
- edit Docs/Drive files
- broad full-account mirroring

### Capture Posture

Google ingestion should be:

- selective
- provenance-rich
- event-driven where appropriate
- bounded for historical import

Not:

- every message
- every event
- every file
- every change forever

### Auth / Policy Direction

Google source-read relies on:

- source-provider OAuth grants
- source-specific scopes
- tenant-bound encrypted token storage

This session should preserve that posture while keeping source-read distinct
from memory-only access.

---

## Source Rules

### Gmail

Default posture:

- event-driven
- selective thread capture
- preserve source references

Recommended rules:

- continue using meaningful-thread heuristics rather than single-message noise
- capture recent thread context, not entire mailbox history by default
- preserve thread/message ids for provenance
- keep mailbox as source of truth

### Calendar

Default posture:

- event-driven
- selective event capture
- preserve source event refs

Recommended rules:

- capture events that matter to memory and context assembly
- avoid overcapturing transient/noisy blocks
- keep calendar authoritative for exact event state

### Drive / Docs

Default posture:

- explicit inclusion first
- narrow folder/tag/metadata cues
- optionally poll or watch targeted locations

Recommended rules:

- preserve explicit inclusion patterns like `/to-brain/` or equivalent metadata
- exclude generated output shelves from re-ingestion
- capture document meaning/provenance, not blind mirror copies

### Historical Import

Historical import remains distinct from live source-read rollout.

Rules:

- bounded
- lower-trust provenance (`bootstrap_import`)
- separate operational lane
- not a substitute for live source-read

## Part 3: Types / Contract

The rollout contract should define shapes along these lines:

```typescript
export type GoogleSourceKind = 'gmail' | 'calendar' | 'drive'

export interface GoogleSourceRef {
  kind: GoogleSourceKind
  sourceId: string
  sourceUrl?: string | null
}

export interface GoogleSourceReadProfile {
  canReadGmail: boolean
  canReadCalendar: boolean
  canReadDrive: boolean
  canWriteGoogle: false
}
```

Exact names may differ, but the contract must answer:

- what source family did this memory come from?
- how does the system point back to the native Google object?
- what can this rollout do and what can it not do?

## Part 5: MCP / Tool Surface

Default recommendation:

- do **not** expose broad Google source-read tools to every external MCP-native
  client in this session
- keep this primarily an ingestion/runtime rollout, not a user-facing broad
  tool explosion

Possible minimal user-facing additions are allowed only if clearly justified and
low-risk, but the default posture should be:

- Google source-read feeds the brain
- the brain is then queried through canonical memory tools

This keeps source complexity behind the brain rather than forcing every client
to reason about Gmail/Calendar/Drive directly.

    while `brain-actions` remains deferred.

- **Migration Decision**
  - No migration was needed.
  - Session 9.5 stayed migration-free by reusing:
    - existing Google OAuth/token metadata storage
    - existing canonical capture/doc/artifact/projection tables
    - existing source-system/source-ref fields

- **Deviations From Planned Scope**
  - The spec pointed toward source-specific event-driven fetches from native
    object identifiers. In the live repo, the existing Gmail and Calendar
    webhook handlers were only receiving provider trigger metadata, not stable
    thread/event ids. Rather than broaden 9.5 into sync-cursor state or mailbox
    mirroring, the shipped implementation uses the smallest safe bounded
    refresh:
    - Gmail webhook trigger -> recent-thread selective refresh
    - Calendar webhook trigger -> recent-updated-event selective refresh
  - No new public MCP tool names were added for Google source-read. The rollout
    remains primarily an ingestion/runtime lane behind the canonical brain.
  - Drive / Docs shipped as explicit inclusion service support plus provenance
    readback; it did not broaden into a new public HTTP route or a generalized
    whole-account poller.
